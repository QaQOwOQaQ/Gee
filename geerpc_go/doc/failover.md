# XClient 重试与故障转移说明

`XClient.Call` 现在支持显式开启的客户端重试与故障转移，用来处理连接建立失败、连接失效或节点级异常这类“还没拿到有效业务响应”的错误。

## 设计目标

- 保持默认行为不变
  - 不配置时，`NewXClient(...)` 与之前完全一致，不会自动重试。
- 只增强 `XClient.Call`
  - v1 先把普通调用链路补完整，`CallByKey` 和 `Broadcast` 保持现有语义。
- 只处理连接/节点级失败
  - 重点是提升可用性，而不是把所有错误都自动吞掉重试。

## 配置方式

```go
xc := xclient.NewXClientWithConfig(d, xclient.RoundRobinSelect, nil, &xclient.XClientConfig{
	RetryPolicy: xclient.RetryPolicy{
		MaxRetries: 1,
		Backoff:    0,
	},
})
```

字段含义：

- `MaxRetries`
  - 首次调用失败后，最多再重试多少次。
  - `0` 表示关闭重试。
- `Backoff`
  - 每次重试前的固定等待时间。
  - `0` 表示不等待，立即切换节点。

## 当前重试范围

会触发故障转移的错误：

- 拨号失败
- 连接已关闭或不可用，例如 `ErrShutdown`
- 连接级 EOF / UnexpectedEOF
- 非超时类 `net.Error`

不会触发故障转移的错误：

- `context` 取消
- `context` 超时
- 服务端返回的业务错误
- 服务端处理超时错误

## 故障转移策略

- 每次顶层 `Call` 都会维护一份 `attemptedAddrs`。
- 首次调用仍按当前 discovery 和负载均衡策略选点。
- 进入重试后，会刷新 discovery，并继续按原有策略选点。
- 如果新选中的节点本次已经尝试过，就跳过它，继续找未尝试节点。
- 同一次顶层调用不会重复打同一节点。
- 发现坏连接后，会把该连接从缓存池中移除。

如果当前 discovery 支持 `Done(addr)`，跳过已尝试节点时也会及时归还计数，避免最少连接数这类策略的内部状态被污染。

## 为什么没有给 `CallByKey` 自动接入

`CallByKey` 的核心价值是“稳定业务 key 尽量稳定路由”。如果自动故障转移到别的节点，确实能提升可用性，但也会改变稳定路由语义。

当前版本优先保留一致性哈希的语义边界，因此：

- `Call` 优先可用性
- `CallByKey` 优先稳定性

后续如果要继续演进，可以把 `CallByKey` 的 failover 做成显式开关，而不是默认行为。

## 为什么没有给 `Broadcast` 自动接入

`Broadcast` 本身会对所有节点并发发起调用。它的失败语义和普通单点调用完全不同：

- 是单个节点失败就整体失败，还是允许部分成功？
- 失败后只补发失败节点，还是整轮重发？
- reply 是取第一个成功结果，还是做聚合？

这些问题在 v1 里先不扩展，避免把行为复杂化。

## 统计项

`XClient.Stats()` 新增了四个字段：

- `RetriedCalls`
  - 发生过至少一次重试的顶层调用数。
- `RetryAttempts`
  - 额外重试次数总和，不包含首次调用。
- `FailoverSuccesses`
  - 经重试后最终成功的调用数。
- `FailoverFailures`
  - 进入重试流程但最终仍失败的调用数。

## 使用边界

这套能力适合：

- 查询类 RPC
- 幂等更新
- 明确可安全重试的方法

这套能力不适合直接默认用于：

- 非幂等写操作
- 有严格一次性语义的请求
- 调用方无法接受重复执行副作用的场景

一句话概括：

> 这是“提升可用性”的客户端容灾，不是“严格一次执行保证”。
