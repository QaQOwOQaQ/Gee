Gemini 说
这个程序为你提供了两种非常直观的测试路径。假设你已经把这段代码保存为 main.go（或者放在 cmd/ 目录下），并且你的 Go 环境和本地的 geecache 模块都已经配置完毕，你可以按照以下两种方式来运行它：

1. 快速验证模式（单机 Demo）
这是最简单的测试方法，纯粹在单进程的内存中演示，主要用来验证你的缓存“拦截”逻辑是否生效。

操作步骤：
打开终端，在代码所在的目录下运行：

Bash
go run main.go -demo
预期效果：
程序会自动执行 runDemo() 函数里的流程，你会在终端看到非常清晰的执行轨迹。重点观察带有 [模拟数据库] 的日志：

拦截测试：第一次查询 Tom 时，会打印数据库查询日志（说明缓存未命中，去查底库了）；第二次查询 Tom 时，直接返回结果，没有出现数据库查询日志（说明缓存命中，成功拦截了底层调用）。

容错测试：查询不存在的 Unknown 时，会按预期抛出你定义的错误。

2. 集群实战模式（分布式服务）
这个模式下，你将真正在本地拉起一个由 3 个节点组成的分布式缓存网络，这也是这个程序最核心的价值。

操作步骤：
你需要打开 3 个独立的终端窗口（模拟 3 台不同的物理机），在对应的目录下分别运行以下命令：

终端 1（启动节点 8001）：

Bash
go run main.go -port 8001
终端 2（启动节点 8002）：

Bash
go run main.go -port 8002
终端 3（启动节点 8003）：

Bash
go run main.go -port 8003
如何进行外部访问测试？
这三个节点启动并注册到 pool 后，会处于阻塞监听状态。你需要打开 第 4 个终端，充当真实用户的客户端，向它们发送 HTTP 请求。

(注：如果你使用的是经典的极客兔兔 GeeCache 实现，它的 HTTP 模块默认的路由前缀通常是 /_geecache/<group>/<key>)

你可以使用 curl 命令来进行测试：

Bash
# 向 8001 节点查询 Tom 的分数
curl http://localhost:8001/_geecache/scores/Tom

# 向 8002 节点查询 Alice 的分数
curl http://localhost:8002/_geecache/scores/Alice
观察背后的“魔法”：
在分布式模式下，当你向 8001 请求 Alice 时，如果经过哈希计算发现 Alice 的数据归 8003 节点管，8001 节点并不会报错，而是会在底层作为客户端发起一次 RPC/HTTP 请求去问 8003 要数据，然后再将结果透传给你。你可以通过查看那三个服务端终端的日志，来观察到底是谁真正去请求了 [模拟数据库]。
