# GeeCache

参考自：https://geektutu.com/post/geecache-day7.html

一个基于 Go 语言实现的分布式缓存系统，参考 [groupcache](https://github.com/golang/groupcache) 设计，支持 LRU 淘汰、一致性哈希节点路由、singleflight 防缓存击穿，以及基于 HTTP + Protobuf 的节点间通信。

---

## 核心特性

| 特性                                 | 说明                                                         |
| ------------------------------------ | ------------------------------------------------------------ |
| **LRU 缓存淘汰（冷热分区）**         | 冷热双链表设计，新数据进冷区，二次访问后升热区，解决预读失效和热点污染问题；支持保护窗口防止误升区 |
| **TTL 过期**                         | 每个 Group 可配置统一过期时长，访问时惰性检测，过期自动失效重建 |
| **Cache Aside**                      | 提供 `Delete(key)` 接口，数据库更新后主动删除缓存，保证最终一致性 |
| **并发安全**                         | 每个缓存组通过互斥锁封装，支持多 goroutine 并发读写          |
| **槽机制哈希（Redis Cluster 风格）** | 固定 16384 个槽，O(1) 节点查找，支持动态添加/移除节点并自动均衡 |
| **singleflight**                     | 相同 key 的并发请求只触发一次数据加载，防止缓存击穿          |
| **HTTP 分布式通信**                  | 节点间通过 HTTP 互相查询缓存，支持多节点水平扩展             |
| **Protobuf 编码**                    | 节点间通信使用 protobuf 序列化，减少网络传输开销             |
| **命名空间隔离**                     | 通过 Group 机制隔离不同业务数据（如用户、商品），互不干扰    |

## 架构说明

```
请求进入 Group.Get(key)
        │
        ▼
  本地缓存命中？──是──▶ 返回缓存值
        │否
        ▼
  singleflight.Do     ← 合并相同 key 的并发请求，只执行一次
        │
        ▼
  有远程节点？──是──▶ 一致性哈希选择节点 ──▶ HTTP 请求远程节点
        │否                                        │
        ▼                                         │
  Getter 回调         ◀────────────────────────────┘
  （从数据库/文件等数据源加载）
        │
        ▼
  写入本地缓存，返回结果
```

## 在真实项目中使用

### 场景描述

假设你有一个 Web 服务，频繁查询 MySQL 中的用户信息。为了降低数据库压力，你希望在多台服务器之间共享缓存，避免每台机器都独立查库。

**部署结构：**

```
客户端请求
    │
    ▼
[ Web 服务 A ]  [ Web 服务 B ]  [ Web 服务 C ]
  :8001            :8002            :8003
    │                │                │
    └────────────────┴────────────────┘
           节点间通过 HTTP 互相查询缓存
                       │
                       ▼
                  [ MySQL ]
              （仅缓存未命中时查询）
```

### 第二步：编写节点启动程序

每台机器上运行同一份程序，通过命令行参数区分自身地址：

```go
package main

import (
    "database/sql"
    "flag"
    "fmt"
    "geecache"
    "log"
    "net/http"

    _ "github.com/go-sql-driver/mysql"
)

var (
    // 所有节点地址，生产环境可改为从配置中心读取
    allPeers = []string{
        "http://10.0.0.1:8001",
        "http://10.0.0.2:8002",
        "http://10.0.0.3:8003",
    }
)

func main() {
    // 通过命令行指定当前节点地址，例如：./server -addr http://10.0.0.1:8001
    addr := flag.String("addr", "http://10.0.0.1:8001", "当前节点地址")
    flag.Parse()

    // 连接 MySQL
    db, err := sql.Open("mysql", "user:password@tcp(127.0.0.1:3306)/mydb")
    if err != nil {
        log.Fatal(err)
    }

    // 创建缓存 Group："users" 命名空间，最大内存 64MB，TTL 5 分钟，抖动 ±1 分钟
    // 实际 TTL 在 [4min, 6min] 之间随机分布，防止缓存雪崩
    group := geecache.NewGroup("users", 64<<20, geecache.GetterFunc(
        func(key string) ([]byte, error) {
            // key 为用户 ID
            var name string
            err := db.QueryRow("SELECT name FROM users WHERE id = ?", key).Scan(&name)
            if err != nil {
                return nil, fmt.Errorf("user %s not found: %v", key, err)
            }
            log.Printf("[MySQL] 查询用户 %s", key)
            return []byte(name), nil
        },
    ), 5*time.Minute, 1*time.Minute)

    // 创建 HTTP 节点池，self 为当前节点自身地址
    pool := geecache.NewHTTPPool(*addr)
    pool.Set(allPeers...) // 注册所有节点

    // 将节点池注册到所有 Group
    geecache.GetGroup("users").RegisterPeers(pool)

    // 启动 HTTP 服务（同时承担：对外提供缓存查询 + 节点间通信）
    port := (*addr)[len(*addr)-4:] // 取最后4位作为端口（示意）
    log.Printf("节点 %s 启动，监听端口 %s", *addr, port)
    log.Fatal(http.ListenAndServe("0.0.0.0:"+port, pool))
}
```

### 第三步：在业务代码中查询缓存

```go
func getUserName(userID string) (string, error) {
    group := geecache.GetGroup("users")
    if group == nil {
        return "", fmt.Errorf("缓存 group 未初始化")
    }

    // 自动处理：本地命中 → 远程节点查询 → MySQL 加载，调用方无需关心
    view, err := group.Get(userID)
    if err != nil {
        return "", err
    }
    return view.String(), nil
}

// Cache Aside 写路径：更新数据库后，主动删除缓存
// 下次查询时将自动从 MySQL 重新加载最新数据
func updateUserName(userID, newName string) error {
    // 1. 先更新数据库
    _, err := db.Exec("UPDATE users SET name = ? WHERE id = ?", newName, userID)
    if err != nil {
        return err
    }
    // 2. 再删除缓存（而非更新），避免并发竞争导致脏数据
    group := geecache.GetGroup("users")
    if group != nil {
        group.Delete(userID)
    }
    return nil
}
```

### 注意事项

- **节点列表一致性**：所有节点的 `allPeers` 必须相同，否则一致性哈希路由会错乱
- **内存限制**：`cacheBytes` 根据实际机器内存合理设置，建议不超过可用内存的 50%
- **TTL 与抖动设置**：根据数据更新频率选择合适的 TTL，建议同时设置 `ttlJitter`（推荐为 TTL 的 10%~20%）以分散过期时间，防止缓存雪崩；更新频繁的数据建议设置较短的 TTL，几乎不变的数据可设为 0（永不过期）
- **Cache Aside 写路径**：数据库更新后务必调用 `group.Delete(key)` 删除对应缓存，否则在 TTL 过期前会读到旧数据
- **多 Group**：可以创建多个 Group 隔离不同业务数据，每个 Group 独立管理内存、TTL 和数据源
- **节点扩容**：新增节点需要重启所有节点并更新 `allPeers` 列表（当前实现不支持热更新）

## 核心模块说明

### LRU（`lru/lru.go`）

采用**冷热分区**设计，解决标准 LRU 的两个经典问题：

| 问题         | 标准 LRU                         | 冷热分区 LRU               |
| ------------ | -------------------------------- | -------------------------- |
| 预读失效     | 批量顺序读取会把热点数据挤出缓存 | 新数据只进冷区，不影响热区 |
| 热点数据污染 | 突发访问的数据长期占据缓存       | 需二次访问才能升入热区     |

**分区结构：**

```
热区 warmList（占总容量 63%）：[最近访问] → ... → [较旧]
冷区 coldList（占总容量 37%）：[新插入]   → ... → [最旧]

淘汰顺序：优先淘汰冷区尾部，冷区为空时才淘汰热区尾部
```

**数据流向：**

- `Add(key)`：新数据插入冷区头部，记录进入冷区的时间
- `Get(key)` 冷区命中：若在冷区停留时间超过保护窗口（`coldProtectWindow`）则升入热区，否则仅移到冷区头部
- `Get(key)` 热区命中：移到热区头部

**保护窗口**（参考 MySQL `innodb_old_blocks_time`，默认 1 秒）：
防止批量顺序读取在极短时间内二次访问同一数据，误判为热点而升区。冷区数据必须停留超过保护窗口才允许升入热区。

**其他特性：**

- **读**：O(1) 哈希查找；若已过期则惰性删除并返回未命中
- **写**：O(1) 插入冷区头部；支持 TTL 参数，0 表示永不过期
- **删**：`Remove(key)` 主动删除指定条目，用于 Cache Aside 写路径
- 支持 `OnEvicted` 回调，条目被淘汰或删除时触发

### 一致性哈希（`consistenthash/consistenthash.go`）

参考 Redis Cluster 的**槽机制**实现，将 key 空间划分为固定的 16384 个槽：

- `hash(key) % 16384` 得到槽编号，O(1) 直接查槽归属节点（无需二分查找）
- 节点均分所有槽，每次 `Add`/`Remove` 后自动 `rebalance` 重新分配
- 槽归属显式可控，运维友好，每个槽归属哪个节点一目了然
- 支持 `Remove` 动态下线节点，其槽自动均分给剩余节点

| 操作     | 复杂度   | 说明              |
| -------- | -------- | ----------------- |
| `Get`    | O(1)     | 直接查 slots 数组 |
| `Add`    | O(16384) | 重新均分所有槽    |
| `Remove` | O(16384) | 重新均分所有槽    |

> 扩容时约有 50% 的槽会重新分配（全量 rebalance），比虚拟节点方案迁移量略多，但槽归属更可控。

### singleflight（`singleflight/singleflight.go`）

- 同一个 key 的首个请求会执行数据加载函数
- 后续并发请求通过 `sync.WaitGroup` 阻塞等待，共享同一结果
- 有效防止缓存击穿（大量并发请求同时打到数据库）

# FAQ

## 为什么节点间通信是 HTTP 而不是 RCP？

在这个项目里用 HTTP 做节点间通信，主要是工程取舍，不是做不到 RPC。

  HTTP 的好处在这个项目里很实际：

  - <font color=blue>实现简单。</font>net/http 标准库直接可用，服务端和客户端都很轻，适合这个缓存项目当前的体量。
  - <font color=blue>调试方便。</font>请求路径、状态码、响应体都容易抓包和排查，出问题时比自定义 RPC 更直观。
  - <font color=blue>依赖更少。</font>现在已经用 protobuf 做消息编码了，再套一层复杂 RPC 框架收益不大。
  - <font color=blue>兼容性更好。</font>以后如果别的语言或别的服务要接入，HTTP 天然更通用。
  - 职责清晰。这个项目目前只需要一个很简单的“远程取 key”能力，用 HTTP 已经够了。

  为什么没优先用 RPC：

  - Go 自带 net/rpc 比较老，生态和扩展性都一般，现在很少拿它做新项目核心通信。
  - 如果上 gRPC，确实更规范，但会引入更多工程复杂度：服务定义、连接管理、状态码语义、流控、拦截器、依赖链都更
    重。
  - 这个项目的瓶颈通常不在“HTTP vs RPC”这层，而在缓存语义本身，比如路由、失效、一致性、并发控制。

  一句话说，就是：这个项目当前的通信模型很简单，HTTP + protobuf 已经覆盖需求，成本比 RPC 更低。

## Singleflight（防缓存击穿机制）

**Singleflight** 是一种在并发编程中常用的设计模式和并发控制机制（最著名的是在 Go 语言的 `golang.org/x/sync/singleflight` 标准扩展库中实现）。

它的核心作用是：**<font color=blue>合并相同的并发请求，防止“缓存击穿”或“惊群效应”（Thundering Herd Problem），从而极大地减轻后端系统（如数据库）的压力。</font>**

> 在技术圈内，这个词通常**不直接翻译**，大家在交流时会直接说英文 “singleflight”。如果一定要用中文表达，通常会根据它的实际功能进行**意译**：
>
> - **请求合并** 或 **并发请求合并**（最准确的功能描述）
> - **单次执行**（强调只有一个请求去执行底层操作）
> - **防缓存击穿机制**（从解决的问题角度描述）
> - 程序员日常交流中有时也会带点玩笑口吻简称它为**“单飞”**。

### 它是如何工作的？

当大量并发请求同时去获取同一个数据（通常带有相同的 Key）时，Singleflight 的工作流程如下：

1. <font color=blue>**拦截请求**</font>：它会拦截所有请求，并检查当前是否已经有一个带有相同 Key 的请求正在执行中。
2. <font color=blue>**单一执行**</font>：如果没有，它放行**第一个**请求去执行底层的耗时操作（例如：查询数据库或调用远程 API）。
3. <font color=blue>**阻塞等待**</font>：在第一个请求执行期间，后续所有带有相同 Key 的并发请求都不会去查询数据库，而是被 Singleflight 阻塞挂起，等待第一个请求的结果。
4. <font color=blue>**共享结果**</font>：当第一个请求执行完毕拿到数据后，Singleflight 会将这份数据**共享**给所有正在等待的并发请求，然后同时唤醒它们。

### 生活中的通俗比喻

假设你和 10 个朋友在一个房间里，大家突然都想知道现在外面下没下雨：

- **没有 Singleflight 的情况**：你们 11 个人同时推开门跑出去看，把门口挤得水泄不通（数据库压力剧增）。
- **有 Singleflight 的情况**：你们派出了 1 个人出去看。剩下的 10 个人在房间里等。那个人回来后大喊一声“下雨了！”，于是大家都得到了答案。此时只有 1 个人执行了“出去看”的动作（只查询了一次数据库）。

### 主要应用场景

- **高并发下的缓存未命中（Cache Miss）**：当某个热点数据突然在 Redis 等缓存中过期（缓存失效）时，瞬间会有成千上万的请求直接打向 MySQL 数据库。Singleflight 可以保证此时只有一个请求去查询 MySQL 并回填缓存，其余请求共享这个结果，完美防止数据库被瞬间打垮。
- **合并相同的外部 API 调用**：如果你的系统需要频繁调用第三方接口，且允许短时间内的请求结果复用，使用它能有效减少网络开销和 API 计费。

## LRU 非线程安全

是一个设计上的职责分离决策，原因有三：

1. <font color=blue>单一职责原则：</font>lru.Cache 只负责 LRU 算法本身（链表操作、哈希查找、淘汰逻辑），锁是并发控制的关注点，不属于算法层。两者混在一起会让代码职责模糊。
2. <font color=blue>锁的粒度由上层决定：</font>cache.go 在 lru.Cache 之上加了一把 sync.Mutex，这把锁的粒度是根据业务需要设计的。如果 LRU 内部也加锁，就会出现双重加锁，要么冗余，要么死锁风险增加。上层比底层更清楚锁应该保护哪个范围。
3. <font color=blue>测试更简单：</font>非线程安全的 lru.Cache 在单元测试中不需要考虑并发时序问题，测试逻辑更清晰、更确定性。并发安全的测试在 cache.go 层单独验证。

 分层结构如下：

```cpp
  lru.Cache          ← 纯算法，不加锁，非线程安全
      ↑
  cache（cache.go）  ← 加 sync.Mutex，线程安全封装
      ↑
  Group（geecache.go）← 业务层，直接使用线程安全的 cache
```

  这种分层的好处是：如果将来要换一种锁（比如读写锁 sync.RWMutex），只需改 cache.go 一处，lru.Cache 完全不用动。                                                                

底层数据结构不加锁也是业界的主流做法。几个典型例子：

* Redis：单线程处理命令，底层数据结构（跳表、哈希表、压缩列表）完全不需要锁。并发安全由事件循环（单线程）保证，而不是数据结构本身。

* Java HashMap vs ConcurrentHashMap：Java 的设计哲学是一样的：

  - HashMap — 不加锁，高性能，单线程或外部加锁使用
  - ConcurrentHashMap — 内部加锁，线程安全，性能略低

  两者分开提供，由使用者根据场景选择，而不是强制所有人都承担锁的开销。

* Go 标准库 container/list：

* lru.Cache 依赖的双向链表本身就是非线程安全的，Go 官方文档明确说明需要调用方自己加锁。标准库里几乎所有容器（map、list、heap）都不加锁。

 总结：底层数据结构追求极致性能，不做并发假设；并发安全交给上层或使用者决定。这样做的核心原因是：锁有性能开销。如果底层强制加锁，在不需要并发的场景（单线程、已有外部锁）下会白白损耗性能。分层设计让两种场景都能以最优方式运行。

## Go 不支持默认参数

  Go 不支持默认参数，这是语言设计上的刻意取舍。Go 的哲学是：函数签名必须显式，不允许隐式省略参数，以避免调用方不清楚实际传入了什么值。

## go.mod、go.sum

在 Go 语言中，`go.mod` 和 `go.sum` 是 Go Modules（Go 的官方依赖管理系统）的两个核心文件。它们通常成对出现在项目的根目录中，共同确保你的项目依赖既准确又安全。

### 1. go.mod：项目的“依赖清单”

- **核心作用**：定义当前项目的模块信息以及它需要哪些外部依赖库来运行。
- **包含的内容**：
  - `module`：当前项目的模块路径（名称）。
  - `go`：项目当前使用的 Go 语言版本。
  - `require`：项目所依赖的第三方包及其特定的语义化版本号（如 `v1.2.3`）。包含直接依赖和部分间接依赖。
  - `exclude` / `replace`：用于排除某个特定版本的包，或者将某个依赖包替换为本地路径或其他远程路径。
- **是否可编辑**：你可以手动编辑它，但更推荐通过 Go 命令行工具（如 `go get`、`go mod tidy`）来自动更新它。
- **类比**：它相当于 Node.js 中的 `package.json`，Java 中的 `pom.xml`，或者 Python 中的 `requirements.txt`。

### 2. go.sum：项目的“安全校验锁”

- **核心作用**：提供依赖包的加密哈希校验，确保代码构建的**一致性**和**安全性**。
- **包含的内容**：记录了项目中所有依赖包（包括直接依赖和所有层级的间接依赖）的特定版本的加密哈希值（通常是 SHA-256 算法生成的指纹）。
- **为什么需要它**：当你或你的团队成员在另一台电脑上拉取代码并下载依赖时，Go 工具链会计算下载下来的依赖包的哈希值，并与 `go.sum` 中的值进行比对。如果一致，说明代码是安全的；如果不一致，说明依赖包可能被篡改（例如遭到供应链攻击）或者服务器上的代码被静默修改了，Go 会拒绝构建并报错。
- **是否可编辑**：**绝对不要手动修改此文件**。它完全由 Go 工具链在背后自动生成和维护。
- **类比**：它相当于 Node.js 中的 `package-lock.json` 或 `yarn.lock`。

------

### 核心区别对比

| 特性               | `go.mod`                           | `go.sum`                                     |
| ------------------ | ---------------------------------- | -------------------------------------------- |
| **中文理解**       | 依赖清单（我要用什么）             | 校验指纹（我用的是不是正版/原版）            |
| **内容详情**       | 模块名、Go 版本、依赖包及版本号    | 所有依赖包及其对应版本的哈希值（哈希校验码） |
| **包含的依赖范围** | 主要记录直接依赖（和部分间接依赖） | 记录 100% 的直接和间接依赖                   |
| **人工干预**       | 可读性高，可手动编辑               | 可读性差，**禁止手动编辑**                   |
| **版本控制**       | **必须**提交到 Git 等代码仓库      | **必须**提交到 Git 等代码仓库                |

**需要注意的一个常见误区**： 这两个文件都**必须提交到你的版本控制系统（如 Git）中**。很多新手会误以为 `go.sum` 是自动生成的就不需要提交，如果不提交 `go.sum`，就失去了构建校验的意义，团队协作时也容易出现环境不一致的问题。

## 为什么用 protobuf 而不用 json？

在这里 protobuf 的作用很明确：它负责节点间传输数据的序列化格式。

  具体说有三点：

  - 定义通信协议。Request 和 Response 在 geecachepb/geecachepb.proto 里固定下来，节点之间都按这个结构收发，避免手写 JSON 字段或自定义字符串协议。
  - 做高效编解码。远程节点请求时把 group、key 编成二进制消息，响应时把缓存值 bytes value 编回来，比 JSON 更紧凑，解析也更省。
  - 保证类型边界清晰。HTTP 只是传输通道，真正“传什么数据”由 protobuf 约束，客户端和服务端都围绕同一份 schema 实现。

  在这个项目里的分工其实是：

  - HTTP：负责把请求发到哪个节点
  - Protobuf：负责请求和响应长什么样、怎么编码
  - GeeCache 自身逻辑：负责 key 路由、缓存命中、回源加载

  如果不用 protobuf，最常见替代是 JSON。那也能工作，但会有这些差别：

  - 包体更大
  - 编解码开销更高
  - []byte 这类字段处理没 protobuf 直接
  - 协议约束没 .proto 明确

  对这个项目来说，protobuf 的核心价值就是：让节点间通信“结构化、紧凑、统一”。

----

主要因为这是内部节点间通信，不是面向浏览器的人类接口，protobuf 更合适。

  核心原因：

  - 更小。protobuf 是二进制编码，请求体和响应体通常比 JSON 更紧凑，节点间频繁传输时更省带宽。
  - 更快。编码和解码开销通常低于 JSON，适合缓存这种高频调用场景。
  - 对二进制数据更自然。这个项目返回的是 bytes value，geecachepb/geecachepb.proto 里直接就是 bytes；如果用 JSON，二进制通常还要做额外转换，表达没那么直接。
  - 协议更严格。.proto 明确定义字段和类型，客户端、服务端共享同一份 schema，不容易出现“字段名写错了还能跑”的隐性问题。
  - 更适合后续演进。以后加字段时，protobuf 的兼容性管理通常比手写 JSON 协议更规范。

----

更准确地说：

  - JSON 是一种通用文本数据格式，重点是可读、通用、方便调试。
  - Protobuf 是一种面向程序之间通信的二进制协议格式，重点是结构约束、体积小、性能高、兼容演进。

  所以你说“功能更多、性能更好”，只对一部分场景成立。

  可以这样理解差别：

  - 对人友好：JSON 更好
  - 对机器友好：Protobuf 更好
  - 临时接口、开放接口：JSON 更常见
  - 内部服务、高频调用：Protobuf 更常见

  protobuf 相比 json 的优势主要是：

  - 更强的 schema 约束，要先定义 .proto
  - 二进制编码，更省流量
  - 编解码通常更快
  - 字段演进更规范，兼容性设计更成熟

  但它也不是全方面“更高级”：

  - 可读性差，抓包看到的是二进制
  - 调试没 JSON 直观
  - 接入门槛更高，要生成代码、维护 schema
  - 对简单脚本、一次性接口有点重

  所以更合适的表述是：

  protobuf 不是 json 的升级版，而是另一类更偏工程化、偏内部通信的序列化方案；在性能、约束和演进上通常更强，但在可读性和易用性上不如 JSON。

# 介绍

这个项目是我用 Go 实现的一个分布式缓存系统，核心思路参考了 Google 的 
  groupcache，支持多节点部署、自动路由和缓存击穿保护。 

它解决的是高并发场景下数据库压力过大的问题。核心思路是：把热点数据缓存在内存里，多台机器组成一个集群，请求进来先查缓存，缓存没有再查数据库。    

项目有三个我觉得比较有意思的设计：

* 第一个是 LRU 淘汰策略。 我用双向链表加哈希表实现，读和写都是                                                       O(1)。内存超限时自动从链表尾部淘汰最久未使用的数据，同时用互斥锁保证并发安全。
* 第二个是一致性哈希。 集群里有多个节点，一个 key  该去哪个节点取，不能随机，否则每个节点都要存全量数据。一致性哈希让同一个 key 始终路由到同一个节点，而且我引入了虚拟节点解决节点分布不均的问题。扩容加节点时，只有少量 key 需要迁移。
* 第三个是 singleflight。 假设缓存刚过期，同一时刻 1000 个请求同时进来查同一个 key，如果每个请求都去查数据库，数据库会被打垮，这叫缓存击穿。我用 singleflight 解决这个问题——相同 key 
  的并发请求只有第一个真正去查，其余的等待并共享同一个结果。

节点间通信用 HTTP，数据用 protobuf 序列化，比 JSON 更紧凑，减少网络开销。                                           

收获：通过这个项目，我对分布式系统里一些常见问题有了更深的理解，比如数据如何在节点间分片、并发场景下怎么避免重复计算、以及接口设计上怎么做到模块解耦——比如 Getter 和 PeerPicker 都是面向接口的，业务层不依赖具体实现。
