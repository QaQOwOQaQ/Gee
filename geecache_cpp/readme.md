**<font color=red>口述：</font>**

* 这个项目是一个基于 C++ 实现的 <font color=blue>分布式应用内缓存组件</font>，目标是把高频读请求尽量拦在缓存层，减少后端数据源压力。
* 整体上它是一个 <font color=blue>分布式结构</font>，支持 <font color=blue>多节点部署</font>，节点之间通过 <font color=blue>HTTP 协议</font> 进行通信，key 到节点的映射通过 <font color=blue>一致性哈希</font> 和 <font color=blue>虚拟节点</font> 来完成，这样在节点增删时只会有一部分 key 重新分布。
* 读请求会先查 <font color=blue>本地缓存</font>，如果 miss，就根据 key 找 <font color=blue>owner 节点</font>；如果 owner 在远端，就通过 <font color=blue>HTTP</font> 去远端取值，否则就 <font color=blue>本地回源加载</font>。
* 为了处理 <font color=blue>高并发场景</font>，我用 <font color=blue>singleflight</font> 合并同一个 key 的并发 miss，避免 <font color=blue>缓存击穿</font>；本地缓存访问则用 <font color=blue>读写锁</font> 封装，保证并发安全。
* 除此之外，为了减少 <font color=blue>重复跨节点访问</font>，我还实现了一个 <font color=blue>hotCache</font>，用来对远端读回的数据做概率性本地副本回填。
* 在 <font color=blue>一致性</font> 上，这个项目采用的是 <font color=blue>Cache Aside 策略</font>，数据更新后会主动失效本地和远端缓存；针对删除与加载并发导致的 <font color=blue>旧值回填</font> 问题，我加了 <font color=blue>版本校验</font> 来保证回填安全；同时我也在缓存层加入了 <font color=blue>TTL 兜底机制</font>，即使失效通知存在延迟或异常，旧数据也会在过期后自动淘汰。这部分我也写了测试做验证。

# 项目架构

这个项目从结构上看，可以拆成四层：**业务入口层、缓存协调层、本地缓存层、节点通信层**。如果面试里让我快速画，我一般会先画“单节点内部结构”，再画“多节点之间怎么协作”。

### 单节点内部结构

```text
                 业务代码 / HTTP 请求
                         |
                         v
                   +-----------+
                   |   Group   |
                   | Get/Set   |
                   | Invalidate|
                   +-----------+
                    |    |    |
        读路径查缓存 |    |    | 写路径失效广播
                    |    |    |
                    v    |    v
              +-----------+   +------------------+
              | mainCache |   |   PeerPicker /   |
              | hotCache  |   | PeerGetter/HTTP  |
              +-----------+   +------------------+
                    |                 |
                    | miss            | 远端 owner 通信
                    v                 v
              +------------------------------+
              | singleflight + version guard |
              +------------------------------+
                             |
                             v
                    +----------------+
                    | getter / setter|
                    | DB / 上游服务  |
                    +----------------+
```

可以这样理解：

- **Group** 是整个缓存组件的业务入口，负责组织读、写、失效三条主链路。
- **mainCache** 是本节点主缓存，通常存本节点负责加载或本地回源的数据。
- **hotCache** 是热点副本缓存，主要存从远端 owner 拉回来的热点数据，减少重复跨节点访问。
- **singleflight** 用来做同 key 并发 miss 合并，避免缓存击穿。
- **version guard** 用来防止删除缓存和加载旧值并发时，旧值回填污染缓存。
- **getter/setter** 代表底层真值源访问逻辑，通常对应 DB、文件、上游服务等慢路径。
- **PeerPicker/PeerGetter/HTTPPool** 负责节点路由、远端访问和健康检查。

### 多节点协作结构

```text
                    +----------------------+
                    |  Consistent Hash Ring |
                    +----------------------+
                      /         |         \
                     /          |          \
                    v           v           v
               +--------+   +--------+   +--------+
               | Node A |   | Node B |   | Node C |
               | Group  |   | Group  |   | Group  |
               | Cache  |   | Cache  |   | Cache  |
               +--------+   +--------+   +--------+
                    \           |           /
                     \          |          /
                      +------ HTTP ------+
```

每个节点本地都有一套完整的缓存组件，但不是各管各的。它们通过两件事协同：

- **一致性哈希**：决定某个 key 的 owner 节点是谁。
- **HTTP peer 通信**：让非 owner 节点在本地 miss 时去 owner 节点取值，或者做失效广播。

所以这个项目的核心不是“把请求平均分给所有机器”，而是：<font color=blue>让同一个 key 尽量稳定落到同一个 owner 节点上，由 owner 负责主加载和主缓存，其他节点优先复用它的结果。</font>

### 读请求数据流

一次 `Get(key)` 的链路可以拆成三条路径：

**1. 本地命中路径**

```text
请求 -> Group::Get -> mainCache / hotCache 命中 -> 直接返回
```

这是最快路径，没有网络，也不会访问后端。

**2. 远端 owner 路径**

```text
请求 -> Group::Get
     -> 本地缓存 miss
     -> 一致性哈希找到 owner
     -> 通过 HTTP 请求远端 owner
     -> owner 查自己的本地缓存 / 必要时回源
     -> 返回结果
     -> 当前节点可能把结果回填到 hotCache
```

这条路径的设计目标是：<font color=blue>避免每个节点在 miss 时都各自回源，而是让 owner 节点统一负责这个 key 的加载。</font>

**3. 本地回源路径**

```text
请求 -> Group::Get
     -> 本地缓存 miss
     -> 选不到可用 owner，或者远端访问失败
     -> 当前节点执行 getter 回源
     -> 写入 mainCache
     -> 返回结果
```

这里的“回源”不是去拿其他 peer 的本地数据，而是当前节点自己访问底层真值源。这个降级是否成功，取决于各节点是否共享同一个后端数据源。

### 写请求与失效链路

这个项目写路径采用的是 **Cache Aside** 思路：

```text
Set(key, value)
 -> setter 先更新底层数据源
 -> 本地删除 mainCache / hotCache
 -> 广播其他 peer 删除这个 key
```

这里没有做“直接更新所有节点上的缓存副本”，而是选择“更新后失效”，原因是：

- 分布式环境下很难确定一个 key 在哪些节点上有副本。
- 直接更新缓存会变成分布式双写，顺序和失败处理都很复杂。
- 删除比更新更幂等，更适合 cache-aside。

### 并发控制与一致性保护

这个项目虽然整体是一套**最终一致性**方案，但在单 key 并发场景里做了两层很关键的保护：

**1. singleflight 防缓存击穿**

同一个 key 并发 miss 时，只允许一个线程真正去远端或后端加载，其余线程等待这个结果返回。这样可以避免热点 key 在高并发下把 DB 或远端 owner 打爆。

**2. version guard 防旧值回填**

写路径失效和读路径加载如果并发交错，可能出现这样的情况：

- 线程 A 已经拿到了旧值，但还没回填缓存
- 线程 B 更新了底层数据，并发起 Invalidate
- 如果没有保护，线程 A 可能把旧值重新写回缓存

所以这里引入了 `versions_` 版本表。失效时推进 version，加载完成回填前校验 version 是否仍然匹配；如果发现期间发生过更新，就拒绝回填旧值。

### hotCache 的定位

`hotCache` 不是主缓存，也不是正式副本机制，它更像一个**近端热点副本层**。

它的主要作用有两个：

- 降低非 owner 节点反复访问远端 owner 的网络开销。
- 在 owner 短时间故障时，给当前节点提供一定的读降级能力。

但它也有边界：

- 容量有限
- 不是所有远端值都会被回填
- 更新后同样需要失效
- 不能替代正式多副本或故障接管机制

### 健康检查与故障退化

节点之间通过 `healthz` 探针做健康检查。某个 peer 被判定不健康后，后续请求不会再优先路由到它，而是退化成：

- 本地 `mainCache` / `hotCache` 命中
- 或本地 `getter` 回源

所以这个项目的故障处理思路不是“强一致切主”，而是：<font color=blue>先绕开故障节点，再尽量本地命中或本地回源，保证系统可继续服务。</font>

### 一句话概括架构

如果让我用一句话总结这套架构，我会这么说：

<font color=24b3aa>这是一个以 Group 为业务入口、以一致性哈希决定 owner、以 HTTP 做 peer 协作、以 mainCache/hotCache 做本地存储、以 singleflight 和版本校验保证高并发正确性的分布式应用内缓存组件。</font>

### 整体架构思路总结

我的整体思路是：尽量用轻量方案解决核心问题，不追求复杂的强一致架构。而是在最终一致性的前提下，把**并发安全**、**路由清晰**和**热点性能**这几个关键点做好。

具体来说，我在架构上重点解决了五个问题：

- **key 归谁负责**：用一致性哈希 + 虚拟节点，让同一个 key 稳定映射到 owner，降低节点增删时的数据抖动。
- **本地 miss 怎么办**：先找 owner 节点拿；owner 不可用时再退化到本地回源。
- **热点 key 重复跨节点怎么办**：通过 hotCache 做近端热点副本，减少反复走网络。
- **高并发 miss 怎么办**：用 singleflight 合并同 key 的并发加载，防止缓存击穿。
- **更新和加载并发怎么办**：用 Cache Aside 做写路径，用版本校验阻止旧值回填。

所以它不是一个追求强一致、自动切主、多副本接管的完整分布式缓存系统；而是一套更强调**工程可实现性、性能收益和关键并发正确性**的分布式应用内缓存方案。

# FAQ

## 分布式应用内缓存

分布式应用内缓存”可以拆成三部分理解：应用内、缓存、分布式。

<font color=blue>先说“应用内”。</font>

它不是像 Redis 这种独立部署的外部缓存服务，而是直接嵌在业务进程里的一个缓存模块。**业务代码调用的是这个组件暴露出来的接口，比如这里的Group::Get、Set、Invalidate，而不是通过一个单独的缓存中间件去访问**，核心入口在 src/geecache.h:68。

<font color=blue>再说“缓存”。</font>

它本质上是把后端数据源里的热点数据先放到内存里。后续请求来了，优先从内存取；只有没命中时，才去回源加载。这个项目里本地缓存是 main_cache_ 和 hot_cache_，读路径在 src/geecache.cpp:145。

  <font color=blue>最后说“分布式”。</font>

这里的意思不是单机里做一个缓存，而是 **多个应用节点上的本地缓存会协同工作**。每个 key 都通过一致性哈希分配一个 owner 节点，请求打到任意节点时，如果这个 key 不归当前节点负责，当前节点会转去 owner 节点拿数据，而不是所有节点各自回源。对应的 peer 抽象在 src/peers.h:12，路由和 HTTP 通信在 src/http.cpp:71。

  所以更准确地说，这个项目不是“一个分布式缓存服务”，而是“一个部署在多个业务节点内部、每个节点有本地缓存、节点之间通过路由和通信协同工作的缓存组件”。

> 我用一个很具体的业务场景讲，你会更容易理解。
>
> 假设你在做一个电商系统，有 3 台应用服务器：
>
>   - 节点 A
>   - 节点 B
>   - 节点 C
>
>   现在有很多请求在查商品详情，比如查商品 item:1001 的标题、价格、库存这些信息。真实数据在数据库里，但如果每次请求都直接打数据库，压力会很大，所以我们想加缓存。
>
> 如果只是“普通应用内缓存”，那每台机器都会自己在内存里放一份缓存，各管各的：
>
>  - 请求打到 A，A 查自己的内存
>   - 请求打到 B，B 查自己的内存
>   - 请求打到 C，C 查自己的内存
>
>  这样做的问题是：
>
>     1. 同一个 key 可能在三台机器上各缓存一份，比较浪费内存。
>      2. 如果 A 没命中，它会自己查数据库；B 没命中，也会自己查数据库。高并发下，多个节点可能重复回源。
>      3. 数据更新后，你要让三台机器都把旧缓存删掉，否则会不一致。
>
>  而“分布式应用内缓存组件”做的事情，是让这些机器上的本地缓存不再彼此独立，而是协同起来。
>
>  比如这个项目里，会先用一致性哈希决定：
>
>  - item:1001 归节点 B 负责
>   - item:2002 归节点 C 负责
>   - item:3003 归节点 A 负责
>
>  也就是说，每个 key 都有一个 owner 节点。
>
>  这时候如果一个用户请求打到了节点 A，要查 item:1001，整个过程会是这样：
>
>  第一步，A 先查自己的本地缓存。如果 A 本地已经有，就直接返回，这是最快的。
>
>   第二步，如果 A 本地没有，它不会立刻查数据库，而是先算一下这个 key 的 owner 是谁。发现 item:1001 的 owner 是 B。
>
>   第三步，A 就去请求 B。B 作为 owner，会先查自己的缓存；如果 B 也没有，再由 B 去查数据库，把结果放到自己的主缓存里，然后返回给 A。
>
>  这样做的好处是：不是每个节点都各自回源，而是尽量由 owner 统一负责这个 key 的加载。
>
>   你可以把它理解成：
>
>   “每个节点都有本地缓存，但整个集群里对同一个 key 有一个主负责节点，其他节点优先找它，而不是大家一起打数据库。”
>
>  这就是“分布式应用内缓存”的核心意思。
>
>  再结合这个项目里的一个细节讲，会更具体。
>
>  假设之后又有很多请求打到 A，还是查 item:1001。如果 A 每次都去 B 拿，也会多一次网络开销。所以这个项目又做了一个 hotCache 机制：A 从 B 成功拿到数据后，有一定概率把这份远端数据顺手存到自己的本地副本缓存里。这样下次同样的请求再打到 A，就可能直接命中，不用再跨节点访问 B。
>
>  所以这个项目的读链路，本质上是：“先查本地，如果本地没有，就找 owner；owner 没有再回源；远端读回来的值还可能顺手在本地留个副本。”
>
>  再说一个“为什么要分布式”的更新场景。
>
>   假设商品 item:1001 价格从 99 改成 109。如果只是每台机器各自缓存，你就得想办法把 A、B、C 上的旧值都删掉，否则有的用户看到 99，有的看到 109。
>
>   这个项目的做法是，更新后通过 Invalidate 发失效广播：
>
>   - 当前节点先删自己
>  - 再通知其他 peer 把这个 key 也删掉
>
>  这样整个集群的旧缓存会一起失效。对应逻辑在 src/geecache.cpp:93。

## 常见的负载均衡策略有哪些？

负载均衡本质上是把请求或数据分配到多个节点上，避免某一个节点成为瓶颈，同时提升整体吞吐、可用性和扩展性。

不过对我这个项目来说，更准确地说，它做的不只是传统意义上的“请求负载均衡”，而是 <font color=red>“基于 key 的数据路由和分片”</font>。<font color=blue>因为缓存场景里最重要的不是把请求平均打散，而是让同一个 key 尽量稳定地落到同一个 owner 节点上，这样才能保证缓存命中率和数据局部性。</font>

这个项目里的做法是基于一致性哈希加虚拟节点。

  - 每个真实节点会被映射成多个虚拟节点，放到哈希环上。
  - 读请求进来后，会对 key 做哈希，顺时针找到第一个命中的节点，把这个节点当成 owner。
  - 如果 owner 是本机，就本地查主缓存或本地回源。
  - 如果 owner 是远端，就通过 HTTP 去对应节点取值。
  - 另外我还加了健康检查，如果某个 peer 被判定为不健康，路由阶段会暂时跳过，不再继续向它转发请求。

我选择这种策略，主要有三个原因。

第一，它适合分布式缓存这种“按 key 分片”的场景。像轮询这类传统负载均衡适合无状态请求，但缓存不是无状态的。同一个 key 如果今天打到 A，明天打到 B，就会导致缓存局部性变差，命中率下降，甚至需要全量复制数据，成本很高。

  第二，一致性哈希在节点扩缩容时重映射范围比较小。比如增加一个节点，不需要把所有 key 重新分布，只会影响哈希环上相邻的一部分 key。这样可以减少缓存抖动，避免扩容时大量 key 同时失效、回源，把后端打爆。

  第三，虚拟节点能改善数据分布不均的问题。
  如果只有真实节点直接上环，可能会出现某些节点负责的区间特别大，负载不均。加了虚拟节点以后，key 分布通常会更均匀一些。

如果面试官继续追问“为什么这叫负载均衡”，我会补一句：严格来说，这里更偏<font color=blue>“分布式路由策略”</font>，因为它不是把每一个请求平均分给后端，而是把 key 稳定映射到 owner 节点。它的目标是兼顾负载分散和缓存命中，而不是单纯平均流量。

常见的负载均衡策略大概有这些：

  - 轮询 Round Robin
    每个请求依次分发，简单，适合无状态服务。
  - 加权轮询 Weighted Round Robin
    给性能更强的机器更高权重。
  - 随机 / 加权随机
    实现简单，但对长连接或局部热点控制一般。
  - 最少连接 Least Connections
    把新请求发给当前连接数最少的节点，适合连接时长差异大的场景。
  - 最短响应时间 / 最小延迟
    更关注实时负载和响应表现。
  - IP Hash
    让同一个客户端稳定落到同一台机器，常用于会话保持。
  - 一致性哈希 Consistent Hash
    常用于缓存、分布式存储、分片路由。
  - Rendezvous Hash
    也是一种分布式 key 路由方法，和一致性哈希相比在某些实现上更简单。

## Redis 和你这里的应用内缓存有什么区别？

我们这里实现的是 <font color=blue>应用内缓存组件</font>，而 Redis 通常算 <font color=blue>“外缓存”服务</font>，也就是外部缓存。原因很简单：它不是嵌在你的业务进程内部的，而是一个 <font color=blue>独立运行的服务</font>。你的应用要通过网络去连 Redis，读写缓存数据。

所以两者区别可以这样理解：

应用内缓存：

  - 缓存就在业务进程内存里
  - 访问不需要网络
  - 速度更快
  - 但每个节点通常都有自己的一份，管理一致性更复杂

  外部缓存，比如 Redis：

  - 缓存是独立部署的服务
  - 应用通过 TCP 网络访问
  - 所有业务节点共享同一个缓存系统
  - 更容易统一管理、做持久化、做淘汰策略和高可用

  所以 Redis 不是“应用内缓存”，而是“分布式外部缓存系统”。

>  我可以理解为，我这里实现的是一个内部分布式缓存组件，而redis是一个独立的分布式缓存服务？
>
>
>  • 可以，这样理解基本没问题。
>
>  更准确一点可以说：
>
>  你的项目实现的是一个“分布式协作的应用内缓存组件”。
>
>  它的缓存实体是在各个业务节点内部的，作为业务进程的一部分存在；只是这些节点之间又通过一致性哈希、peer 通信和失效广播协同工作，所以它不是单机缓存，而是“应用内 + 分布式协作”。
>
>  而 Redis 更准确的定位是“独立部署的外部缓存服务”。
>
>  它运行在业务进程之外，应用需要通过网络去访问它。至于 Redis 自身能不能做集群、分片、高可用，那是它作为缓存服务的部署形态；但对业务来
>  说，它首先是一个外部服务，而不是应用内组件。

## 既然 Redis 更成熟，为什么你这个项目还要做应用内缓存？

这个问题面试里很常见，核心不是说“我做的比 Redis 好”，而是说清楚两者解决的问题不完全一样。

最稳的回答是：

“不是要替代 Redis，而是两者关注点不同。<font color=blue>Redis 是通用的外部缓存服务，能力完整、生态成熟；我这个项目做的是应用内缓存组件，重点是把高频读请求尽量在业务节点内部就消化掉，减少一次网络访问，同时让多个节点之间对 key 的归属和失效行为有一致的协作机制。</font>”

你可以继续展开成三点。

第一，应用内缓存的访问路径更短。如果请求命中本地缓存，数据就在当前进程内存里，不需要再发一次网络请求到 Redis。对于特别高频、特别小的数据，这一跳网络本身就是成本。所以应用内缓存的优势是极致低延迟。

第二，某些热点数据离业务更近。如果一个业务节点反复访问同一批 key，把这些数据放在本地会更划算。这个项目里除了主缓存，还做了 hotCache，就是想减少非 owner 节点反复跨节点访问的开销。也就是说，它更强调“把热点尽量留在离请求最近的地方”。

第三，<font color=blue>这个项目的重点其实是机制验证，不是工业替代。</font>我做它不是因为 Redis 不成熟，而是因为我想把分布式缓存里几个核心问题自己走通，包括：

  - 一致性哈希路由
  - 同 key 并发 miss 的击穿防护
  - 分布式失效广播
  - 删除与加载并发时的旧值回填问题

这些问题在 Redis 生态里当然也有成熟方案，但自己实现一遍，才能真正理解缓存系统在分布式和并发场景下的关键矛盾。

如果面试里想显得更成熟一点，你可以主动补一句 tradeoff：

“如果是生产环境，我不会用它去替代 Redis。更现实的做法通常是两级缓存:本地应用内缓存做第一层，Redis 做第二层共享缓存。这样既能利用本地命中的低延迟，也能利用 Redis 的统一管理能力。”

## 那应用内缓存相比 Redis，最大的缺点是什么？

因为应用内缓存是分散在每个业务节点里的，不像 Redis 那样是一个相对中心化的外部缓存服务。这会带来几个直接问题：

<font color=blue>第一，同一个 key 可能出现在多个节点上（hotCache、mainCache）。</font>一旦底层数据更新了，你就要想办法把所有节点上的旧值一起失效，否则不同节点可能读到不一样的数据。这也是我这个项目里专门做分布式失效广播，以及版本校验防旧值回填的原因。

<font color=blue>第二，容量是分散的，不容易统一利用。</font>Redis 的内存池是共享的，所有业务节点都访问同一套缓存资源；而应用内缓存是“每台机器各拿自己那部分内存”，可能出现有的节点缓存很满，有的节点缓存很空，整体利用率不如集中式缓存灵活。

<font color=blue>第三，可观测性和运维能力通常不如 Redis。</font>Redis 天然就有比较成熟的持久化、高可用、监控、淘汰、运维工具链；应用内缓存如果要做到这些，通常都得自己补

## 你这个项目是实现了什么级别的一致性？

如果按分布式系统里比较严格的定义来讲，我这个项目实现的是 **最终一致性**，不是 **强一致性**，也不是 **线性一致性**。

我这么判断，主要有几个原因。

第一，写路径采用的是 **Cache Aside**。
<font color=blue>也就是先更新后端数据源，再把本地和远端缓存失效掉。这样做的好处是实现简单、性能好，但代价就是在失效广播延迟、失败，或者某些节点还没来得及重新加载新值的时候，**短时间内仍然可能读到旧值**。</font>所以它本质上不是“所有节点立刻看到同一份最新数据”的模型。

第二，这个项目里的失效广播是 **best-effort** 的。
<font color=blue>如果有些 peer 失效成功了，有些失败了，系统会把错误返回出来，但不会做分布式事务，也不会回滚本地操作，更没有 quorum 或 consensus 机制去保证所有节点同时完成更新。所以从系统级别看，它不可能是强一致。</font>

第三，我还专门加了 **TTL 兜底**。
<font color=blue>TTL 这个设计本身也说明我们接受“某段时间内可能存在旧值”，</font>只是希望即使失效通知异常，旧数据也能在过期后被淘汰。
如果一个系统本来就是强一致的，通常不会把 TTL 当作一致性的兜底手段。

不过，虽然它不是强一致，我还是在**单 key 的并发场景**里做了一层更强的保护：我加了**版本校验**，用来解决“删除缓存”和“加载旧值”并发时的**旧值回填**问题。也就是说，某个 key 在失效之后，不会因为一个已经在路上的旧加载结果，又把旧数据重新写回缓存。
所以更准确一点说：

- **系统整体**：是最终一致性。
- **单 key 本地并发更新/加载交错场景**：我额外保证了不会发生明显的旧值回填错误。

如果这是面试场景，我会用一句更完整的话收住：

## 某个节点宕机了，会发生什么？

如果结合这个项目当前的实现来回答，**某个节点宕机后，不会触发强一致切主或自动数据迁移**，系统的表现更接近“发现故障后绕开该节点，尽量继续服务”，具体会分几种情况。

先说最核心的结论：

- **系统不会因为一个节点挂掉整体不可用**
- **但也不能保证所有 key 都还能正常读到**
- **是否还能继续命中，取决于本地有没有 hotCache、副本是否还在、以及当前节点能不能自己回源**

我会这样展开说。

<font color=red>1. 宕机节点如果是远端 peer，会先被健康检查摘掉</font>

<font color=blue>项目里 HTTPPool 有后台健康检查线程，会周期性探测 peer 的 healthz 接口。</font>一旦某个节点已经被确认存活过，后续探测失败，就会把它标成不健康。这样在 PickPeer 时，就不会再把请求路由到这个节点。

所以从路由层面看，宕机节点会被 **逐步摘除**，而不是一直重试打过去。

<font color=red>2. 在被摘除之前，可能会先经历一次请求失败</font>

如果节点是刚宕机，健康检查还没来得及更新状态，这时某次请求仍然可能被路由过去。这种情况下 HTTP 请求会失败，当前节点会把这个 peer 标成不健康，然后这次读取会退化处理。

对应到实现上，就是：

- <font color=blue>远端请求失败后会更新 peer 健康状态</font>
- 后续同一个节点就不会再被选中

<font color=red>3. 退化后能不能继续读成功，要看数据是不是还能从别处拿到</font>

这里分两种情况。

**情况 A：本地已经有 hotCache**

如果这个 key 之前从远端读成功过，并且已经被概率性回填到本地 hotCache，那么即使 owner 节点挂了，这个节点仍然可能继续从本地热点副本返回数据。

<font color=blue>这也是 hotCache 的一个实际价值：它不只是减少跨节点访问，也能在短时间内提高 owner 宕机时的读可用性。</font>

但要注意，这种可用性是**临时的**，因为：

- hotCache 容量有限
- 有 TTL
- 也不是所有远端数据都会被回填

所以它不是正式副本机制，更像“读降级缓冲”。

**情况 B：本地没有 hotCache**

这时 PickPeer 选不到健康远端节点，代码会退回本地 getter 回源。

但这里是否真的能成功，取决于你的部署模型：

- 如果每个节点都能访问同一个共享后端数据源，比如共享 MySQL，那当前节点仍然可以自己回源成功。
- 如果每个节点只持有自己那部分数据，而当前节点并不能访问宕机节点对应的数据，那这次读取就会失败。

也就是说，这个项目**没有做多副本存储**，所以 owner 挂掉以后，不是天然还能无损接管。

<font color=red>4. 写和失效广播在节点宕机时也会受影响</font>

如果某个节点挂了，而这时系统发生了数据更新：

- 本地节点会先失效自己的缓存
- 然后广播失效给其他 peer
- 挂掉的节点自然收不到这次失效通知

这里当前实现是 **best-effort 广播**，失败会返回错误，但不会做分布式事务补偿。

所以这会带来一个结果：

- <font color=blue>如果宕机节点之后恢复了，而它本地还保留旧缓存，就可能存在脏数据风险</font>

<font color=blue>不过项目里又加了 TTL，所以即使失效广播漏掉了，这些旧值也不会永久存在，只是会在一段时间内存在旧值窗口。</font>

<font color=red>5. 这个项目没有自动做节点恢复后的重平衡和数据修复</font>

这一点面试里最好主动说出来，因为这能体现你知道系统边界。

当前实现里有：

- 一致性哈希
- 健康检查
- 动态更新 peer 列表

但没有：

- 多副本
- 自动主从切换
- 宕机恢复后的缓存重建协议
- 基于共识的强一致协调

所以它更像一个**高性能、工程化的分布式应用内缓存组件**，而不是一个具备高可用强一致能力的完整分布式缓存集群。

如果我在面试里简洁回答，我会这样说：<font color=24b3aa>某个节点宕机后，这个项目不会做强一致切换，而是通过健康检查把故障节点从路由里摘掉，后续请求不再转发过去。如果本地之前已经有这个 key 的 hotCache 副本，还可以继续命中；如果没有，就会退回本地回源。至于回源能不能成功，要看当前节点是否能访问共享后端数据源。因为这个项目没有做多副本和自动接管，所以它能做到的是故障绕行和一定程度的读降级，而不是完整高可用容灾。失效广播方面，如果节点宕机时没收到失效，恢复后可能短时间持有旧值，但 TTL 会作为兜底逐步淘汰这些旧数据。</font>

## 这个项目算高可用吗？

严格来说，**这个项目不能算“完整意义上的高可用系统”**，但它**具备一定的容错和降级能力**。

我会更稳地把它定义成：

**一个 <font color=blue>具备基础故障绕行能力的分布式应用内缓存组件</font>，而不是一个 <font color=blue>成熟的高可用分布式缓存集群</font>。**

为什么这么说，可以分成两面看。

**它具备的“可用性增强”能力**

- <font color=blue>有 **健康检查**</font>，节点挂掉后，其他节点会逐步把它标记为不健康，后续不再优先路由过去。
- <font color=blue>有 **hotCache**</font>，如果某个远端 key 之前已经被回填到本地热点副本，那么 owner 挂掉后，短时间内仍然可能继续命中。
- <font color=blue>有 **本地回源兜底**</font>，如果选不到健康 peer，当前节点会退回本地 getter 回源；在“所有节点都能访问共享后端数据源”的前提下，请求不一定失败。
- <font color=blue>有 **TTL 兜底**</font>，即使某些失效广播丢了，旧缓存也不会永久存在。

这些能力说明它不是“一个节点挂了就完全不可用”的系统。

**但它还不算高可用的关键原因**

- **<font color=blue>没有多副本机制</font>**。
  owner 节点挂了以后，并没有一个正式的副本节点立刻接管这部分 key。
- **<font color=blue>没有自动故障切换</font>**。
  系统不会重新选主，也不会把某个节点提升成新的 owner。
- **<font color=blue>没有强一致协调机制</font>**。
  没有 quorum、没有共识协议、没有分布式事务，所以做不到强一致下的高可用。
- **<font color=blue>没有自动成员发现和自动重平衡</font>**。
  当前 peer 列表本质上还是配置驱动的，不是完整的服务发现体系。
- **<font color=blue>失效广播是 best-effort</font>**。
  某些节点广播失败后，只能依赖后续重试、再次失效或 TTL 慢慢收敛。
- **<font color=blue>可用性依赖部署前提</font>**。
  如果当前节点自己也无法访问底层数据源，那么 owner 一挂、又没有 hotCache 时，读请求就可能直接失败。

所以更准确地说：

- 如果你把“高可用”理解成“有一定故障容忍和降级能力”，那它**有一些高可用雏形**。
- 如果你把“高可用”理解成“节点故障后系统仍能稳定接管、持续服务、数据语义明确”，那它**还不算**。

## Mysql、Redis、内缓存间的数据一致性

这三者里，DB 必须是唯一真值源，Redis 是共享二级缓存，我们这个分布式应用内缓存组件是一级近端缓存。

现实里通常追求的是<font color=blue>“最终一致性 + 尽量缩短脏数据窗口”</font>，而不是严格强一致。因为一旦你有两层缓存，再加数据库，完全强一致的代价会很高，基本要上分布式事务或串行化写路径，不划算。

推荐分层

  - DB：唯一真值
  - Redis：共享缓存，容量更大，给所有应用节点复用
  - 本地分布式缓存组件：每个应用节点上的近端缓存，用来挡住高频读、降低 Redis 压力和网络跳数

读路径建议走这条链路：

本地缓存组件 -> Redis -> DB

具体是：

    1. 先查本地缓存组件
    2. 本地 miss 后查 Redis
    3. Redis miss 再查 DB
    4. 查到后先回填 Redis，再回填本地缓存组件

这样本地缓存是最快的一层，Redis 是共享兜底层，DB 只在双 miss 时访问。

写路径

这部分是核心。最稳妥的是 Cache Aside：

    1. 先更新 DB
    2. DB 提交成功后，删除 Redis
    3. 再广播失效本地缓存组件

这个顺序很重要，原因是：

  - 如果你先失效本地缓存，再删 Redis，那么中间有请求进来时：
    - 本地 miss
    - 去 Redis 读到旧值
    - 又把旧值重新回填到本地
  - 所以要先把 Redis 这个共享层清掉，再让本地层失效，避免本地从旧 Redis 里“复活”脏数据

为什么不建议“写 DB 后同时更新两层缓存”？

因为双写很容易出顺序问题：

  - DB 更新成功，但 Redis 更新失败
  - Redis 更新成功，但本地广播失败
  - 并发读正好夹在两个更新中间，读到旧值又回填

所以比起“更新缓存”，工程上更常用的是“更新 DB 后删除缓存”。删除失败可以重试，更新失败则很难判断缓存里到底是什么状态。

但这还不够，必须再加“版本号”。

即使你按上面顺序做，仍然会有 <font color=blue>并发窗口</font>：

  - 线程 A 开始读，准备从 DB/Redis 加载旧值
  - 线程 B 更新 DB，并完成 Redis 删除和本地失效
  - 线程 A 晚一点返回，手里还是旧值
  - 如果它这时直接回填内缓存，就会把旧值重新写回去

  这正是你们项目里已经处理过的问题：<font color=blue>版本校验防旧值回填。</font>

  三层场景里，建议把这个版本概念扩展到 Redis：

  - DB 每条记录带一个 version 或 updated_at
  - Redis 缓存值里同时存 value + version
  - 本地缓存组件里也存 value + version
  - 回填时必须比较版本
  - 如果发现当前 key 的失效版本已经比自己手里的版本新，就拒绝回填

  这样即使有并发旧读返回，也不会把旧值重新灌回 Redis 或本地缓存。

  所以三层一致性的关键不是“删缓存”本身，而是两件事

    1. 正确的写顺序
    2. 用版本号阻止旧值回填

  一个更完整的工程化方案。

如果你想把异常情况也补齐，推荐这样设计：

    1. 写请求进入应用
    2. 事务里更新 DB，同时写一条 outbox/event 事件
    3. 事务提交后，由后台任务或 MQ 消费者执行：
       - 删除 Redis
       - 广播本地缓存失效
    4. 如果失败，持续重试
    5. 读链路上的回填统一带版本校验
    6. Redis 和本地缓存都 <font color=blue>设置 TTL，作为最后兜底</font>

  这里 outbox 或 binlog/CDC 的作用，是解决“DB 已经成功，但缓存失效消息没发出去”的问题。否则你只能依赖接口内同步删除，一旦进程在 DB 提交后崩了，缓存就可能长期脏。

TTL 要不要有？

要有，但 TTL 只是兜底，不是主一致性方案。

建议：

  - 本地缓存 TTL 更短
  - Redis TTL 稍长
  - 主流程仍然依赖“写后失效 + 版本校验”

因为只靠 TTL，会让脏数据窗口变成“TTL 多长就可能脏多久”。

> 我大体明白了，Mysql 数据库作为唯一真值源，当 Mysql 更新时，我们采取的是 Cache Aside 策略，先更新 Mysql，再删除 Redis 上的缓存，然后广播删除本地缓存。
>
> 但这样做会存在一个并发窗口：
>
> 1. 线程 A 读 Mysql/Redis 上的旧值
> 2. 线程 B 修改 Mysql 上的旧值为新值，清空 Redis 和本地缓存
> 3. 线程 A 晚一点返回，手里还是旧值，如果它这时直接回填内缓存，就会把旧值重新写回去
>
> 解决办法是 “版本号校验 + TTL 兜底”，给缓存加一个 TTL 作为兜底，然后再给记录加上一个版本号，每次更新 Mysql 时（增删改）就递增这个版本号，这样我们就可以通过比较版本号来判断当前回填到缓存的值是否是更新的。
>
> 不过这样做有一个成本，那就是数据库和缓存都需要多一个 version/update_at 字段。

## 广播本地缓存失效是如何实现的？

**通常是 <font color=blue>处理写请求的应用节点</font> 来触发广播。**更具体地说：

  - 客户端把更新请求打到某个业务节点
  - 这个节点先更新 MySQL
  - 更新成功后删除 Redis
  - 然后由这个节点调用本地缓存组件的 Invalidate(key)
  - Invalidate(key) 再通过 peer 机制，把失效请求广播给其他节点
  - 每个节点收到后，删除自己机器上的 main_cache / hotCache

所以，广播的发起者不是数据库，也不是 Redis，而是写路径所在的应用节点。

你可以把它理解成：

**“谁负责这次写操作，谁负责在写成功后发起缓存失效通知。”**

这也是最常见的做法，因为：

  - 应用层最清楚这次改了哪个 key
  - 应用层也最容易把 更新 DB -> 删 Redis -> 广播本地失效 串成一个完整流程

  不过这里还有一个现实问题：

  如果 MySQL 不是通过这个应用节点改的，而是被别的系统直接改了怎么办？

  这时应用节点就不知道发生了更新，自然也没法主动广播。常见做法有两类：

    1. 约束所有写都走统一业务服务
       - 最简单
       - 所有更新都经过应用层
       - 那就由应用层统一删 Redis + 广播本地缓存失效
    2. 如果存在旁路写，就用订阅变更的方式触发失效
       - 例如 MQ、binlog、CDC、outbox
       - 数据库一旦发生变更，就由消费端去删 Redis，并广播本地缓存失效

  所以严格来说，答案分两层：

  - 正常场景：发起写请求的应用节点触发广播
  - 存在旁路写场景：由变更事件消费者触发广播，而不是某个在线请求节点直接触发

> 具体的，当前写请求所在节点，会遍历所有远端 peer，并对每个 peer 发一个 HTTP DELETE 请求，通知它删除本地对应的缓存 key。

## 多节点部署如何实现的？

多节点部署其实就是把同一套缓存服务部署在多个节点上，<font color=blue>每个节点都配置同一份 peer 地址列表</font>。<font color=blue>启动后，每个节点会基于这些地址构建一致性哈希环</font>，请求进来后先根据 key 算出 owner 节点，如果 owner 是自己就本地处理，如果是远端就通过 HTTP 去拉取数据。数据更新后，再通过广播失效把各节点上的本地缓存清掉，这样就实现了多节点协作。

  ## 节点之间是如何感知到对方的存在的？

<font color=blue>我们这个项目里，节点之间不是靠注册中心自动发现的，而是每个节点启动时都配置同一份 peer 地址列表。</font>这样每个节点都知道集群里有哪些成员，会本地构建一致性哈希环，并为其他节点建立 HTTP 访问能力，所以请求来了以后就能根据 key 找到 owner 节点，更新时也能把失效广播给其他节点。

也就是说：

  - 节点不会在启动后自动发现彼此
  - 也没有注册中心动态下发节点信息
  - 而是 <font color=blue>启动前就把集群地址列表配好，然后每个节点加载同一份列表</font>

这个项目里，每个节点其实都维护了一份“本地集群视图”：

- 成员是谁：来自 Set(peers)
- key 归谁：来自本地一致性哈希环
- 怎么访问别人：来自 HttpGetter
- 怎么广播失效：来自 GetAllPeers

所以节点之间能协作，不是因为有一个中心在实时协调，而是因为**每个节点在启动时就已经拿到了同一份集群成员信息，并在本地构建了完整的路由和通信能力**。

**边界也要讲清楚**

这套方案的边界是：

- 不支持自动服务发现
- 不会自动感知新节点加入
- 不会自动感知节点永久下线
- 节点成员变化需要外部更新 peer 列表，再调用 Set(peers) 刷新本地视图

具体的，peer 列表是怎么配置，是使用方来决定的，只要使用方把 peer 处理成一个 `verctor<string>` 并发送给节点即可。

> 假设你写了一个 config.yaml 模板，内容是：
>
> ```
> peers:
>     - http://10.0.0.1:8001
>     - http://10.0.0.2:8001
>     - http://10.0.0.3:8001
> ```
>
> 然后给三台机器分别下发：
>
> A 机器：
>
> ```
> self: http://10.0.0.1:8001
> peers:
>     - http://10.0.0.1:8001
>     - http://10.0.0.2:8001
>     - http://10.0.0.3:8001
> ```
>
> B 机器：
>
> ```
> self: http://10.0.0.2:8001
> peers:
>     - http://10.0.0.1:8001
>     - http://10.0.0.2:8001
>     - http://10.0.0.3:8001
> ```
>
> C 机器：
>
> ```
> self: http://10.0.0.3:8001
> peers:
>     - http://10.0.0.1:8001
>     - http://10.0.0.2:8001
>     - http://10.0.0.3:8001
> ```
>
> 程序启动后：
>
>   - A 知道“我自己是 A，集群成员是 A/B/C”
>   - B 知道“我自己是 B，集群成员是 A/B/C”
>   - C 知道“我自己是 C，集群成员是 A/B/C”
>
> 这就完成了“节点之间互相感知”。

程序启动后的流程大致为：

> ```cpp
> auto pool = std::make_shared<geecache::HTTPPool>("http://127.0.0.1:9999");
> pool->Set({"http://127.0.0.1:9999"});
> pool->RegisterGroup(group);
> group->RegisterPeers(pool);
> pool->Start();
> ```
>
> 见 tests/http_test.cpp:47、tests/http_test.cpp:52、src/geecache.h:106。
>
> 下面按职责解释这几行。
>
> 1. **`auto pool = std::make_shared<geecache::HTTPPool>("http://127.0.0.1:9999");`**
>
> * 这一步是在创建“当前节点的网络通信组件”。
> * self 表示“我是谁”，比如："http://10.0.0.1:8001"
> * HTTPPool 内部会记住这个地址，后面有两个用途：
>   * 判断一致性哈希选出来的 owner 是不是自己
>   * 启动本机 HTTP 服务，监听这个地址上的请求
> * 对应构造函数在 src/http.cpp:54。
>
> 你可以把这一步理解成先把当前节点的通信身份建起来。
>
> ---
>
> 2. **`pool->Set(peers);`**
>
>  这一步是在告诉当前节点：“整个集群有哪些成员。”
>
> 比如：
>
> ```cpp
> std::vector<std::string> peers = {
>     "http://10.0.0.1:8001",
>     "http://10.0.0.2:8001",
>     "http://10.0.0.3:8001",
> };
> pool->Set(peers);
> ```
>
>  Set 做了两件核心事情：
>
> * 创建一致性哈希环 peers_
> * 为每个 peer 创建一个 HttpGetter
>
>  源码在 src/http.cpp:61。
>
>  也就是：
>
> ```cpp
> peers_ = std::make_unique<consistenthash::Map>(kDefaultReplicas);
> peers_->Add(peers);
> http_getters_[peer] = std::make_shared<HttpGetter>(peer + base_path_);
> ```
>
> * `peers_->Add(peers)`：把所有节点放进哈希环，后面用来做 key -> owner 路由
>
> * `http_getters_[peer] = ...`：给每个远端节点准备一个 HTTP 客户端，后面可以直接发请求过去
>
>   所以这一步可以理解成：
>
>   把“集群成员表”和“访问其他节点的能力”都准备好。
>
> ----
>
> 3. **`pool->RegisterGroup(group);`**
>
> 这一步很容易漏掉，但它很重要。
>
> 它的作用是：
>
> 把本地这个 group 挂到 HTTP 服务上，允许远端节点通过 HTTP 访问它。
>
> 源码在 src/http.cpp:101。
>
> 因为别的节点访问你时，HTTP 请求路径会长这样：
>
> `/_geecache/<group_name>/<key>`
>
> 收到请求后，HTTPPool 要先根据 group_name 找到本地对应的 Group 实例，再去查缓存或回源。
>
> 如果不 `RegisterGroup(group)`，远端请求打过来时，本机 HTTP 服务都不知道该把请求交给哪个缓存组处理。
>
>   所以这一步是：
>
>   把本地缓存组暴露给 HTTP 服务。
>
> ----
>
>     4. **`group->RegisterPeers(pool);`**
>
> 这一步是反过来：
>
> 把 pool 注册给 group，让 group 具备“访问其他节点”的能力。
>
> 源码在 src/geecache.cpp:158。
>
> 因为 group 在处理读请求时，如果本地 miss，会调用：
>
>   - `peers->PickPeer(key)` 看这个 key 属于哪个 owner
>   - 如果 owner 是远端，就通过 `PeerGetter` 去远端取值
>
> 这段逻辑在 src/geecache.cpp:172。
>
> 也就是说：
>
>   - `RegisterGroup(group)` 是让“别人能访问我”
>   - `RegisterPeers(pool)` 是让“我能访问别人”
>
> 这两个方向别混了。
>
> ---------
>
>     5. **`pool->Start();`**
>
> 这一步是把 HTTP 服务真正启动起来。
>
> 源码在 src/http.cpp:166。
>
> 它会：
>
>   - 注册 `GET /_geecache/...` 路由，用于远端读
>   - 注册 `DELETE /_geecache/...` 路由，用于远端失效
>   - 解析 self 里的 host 和 port
>   - 在后台线程里启动 HTTP server
>
> 也就是说，从这一步开始，这个节点才真正“上线”，能接收其他 peer 的请求。
>
> ---------
>
> 把这 5 步串起来看
>
> 假设当前节点是 A：
>
> ```cpp
> auto pool = std::make_shared<HTTPPool>("http://10.0.0.1:8001");
> pool->Set({
>     "http://10.0.0.1:8001",
>     "http://10.0.0.2:8001",
>     "http://10.0.0.3:8001"
> });
> pool->RegisterGroup(group);
> group->RegisterPeers(pool);
> pool->Start();
> ```
>
>   它的含义就是：
>
>     1. 创建 A 节点自己的通信组件
>     2. 告诉 A 整个集群有 A/B/C
>     3. 把本地缓存组挂到 HTTP 服务上，允许别人来访问 A
>     4. 让本地缓存组知道如何根据 key 去访问 B/C
>     5. 启动 A 的 HTTP 服务，正式开始对外提供 peer 能力
>
> ---------
>
>   你可以把它类比成两次“接线”
>
>   - `pool->RegisterGroup(group)`
>     是把“HTTP 入口”接到“本地缓存逻辑”
>   - `group->RegisterPeers(pool)`
>     是把“本地缓存逻辑”接到“远端节点访问能力”
>
>   所以 group 和 pool 其实是双向配合的：
>
>   - pool 收到远端 HTTP 请求后，会调用 group
>   - group 需要访问远端 peer 时，会调用 pool

## 如果新增或下线节点，这种静态 peer 列表怎么更新？

在这个项目里，静态 peer 列表的更新本质上靠重新下发配置，因为它没有注册中心，也没有动态成员发现。

也就是说，如果新增或下线节点，通常要做这几步。

**<font color=blue>新增节点</font>**

假设原来是 A/B/C，现在要加 D。

  1. 先把配置里的 peer 列表从 A,B,C 改成 A,B,C,D

  2. 把这份新配置下发到所有节点

     也就是 A、B、C、D 都要拿到同一份新列表

  3. <font color=blue>让各节点重新执行类似 pool.Set(peers) 的逻辑，重建一致性哈希环</font>

  4. 新节点 D 启动 HTTP 服务，对外提供访问能力

  更新完成后，所有节点都会认为集群成员是 A/B/C/D，同一个 key 的 owner 计算结果也会按新哈希环变化。

**<font color=blue>下线节点</font>**

假设原来是 A/B/C，现在要下掉 C。

1. 先把 peer 列表从 A,B,C 改成 A,B
2. 把新配置下发到剩余节点
3. 剩余节点重建哈希环
4. 再摘掉 C 的流量并停机

  这里顺序要注意：先更新其余节点配置，再真正停掉节点，否则其他节点还以为 C 存在，请求可能继续转发过去。

**<font color=blue>线上会发生什么？</font>**

不管是新增还是下线，只要 peer 列表变化并重建哈希环，就会有一部分 key 的 owner 发生变化。

这会带来两个结果：

  - 部分请求会被路由到新的 owner
  - 新 owner 一开始本地没数据，会出现一段时间的缓存 miss，需要重新从 Redis/DB 加载

这就是节点变更时常见的 <font color=24b3aa>“缓存抖动”</font> 或 <font color=24b3aa>“局部冷启动”</font>。

一致性哈希的价值就在这里：它不是让 key 完全不变，而是让只有一部分 key 迁移，而不是全量重映射。

所以在当前项目语境下，更稳妥的说法是：

新增或下线节点通常需要修改静态配置，并让所有节点重载配置或重启服务，以保证大家看到的是同一份 peer 列表。

**<font color=blue>工程上更稳的做法</font>**

如果要做线上可用性更强的版本，通常会这样改进：

  - 用配置中心或注册中心统一管理 peer 列表
  - 节点监听成员变更，动态调用 Set(peers) 重建哈希环
  - <font color=24b3aa>下线前先摘流，再更新成员列表</font>
  - <font color=24b3aa>新节点先加入成员表，再做预热，最后接流量</font>

<font color=blue>面试口语版</font>

这个项目里 peer 是静态配置的，所以新增或下线节点时，一般是先修改配置文件里的 peer 列表，再把新配置下发到所有节点，让每个节点重新构建一致性哈希环。新增节点后，会有一部分 key 重新映射到新节点；下线节点时，也会有一部分 key 转移到其他节点。这种方式实现简单，但缺点是成员变更依赖人工配置或统一部署系统，不具备注册中心那种自动发现和自动摘挂能力。”

## 节点间数据路由怎么做的？

在这个项目里，节点间数据路由的核心是：先用一致性哈希算出某个 key 应该归哪个节点负责，再通过 HTTP 去访问那个节点。

假设有 A、B、C 3 个节点，每个节点启动时都会拿到同一份 peer 列表，在本地构建同一个一致性哈希环（见 `src/http.cpp:61`）。当读请求（如查 `user:1001`）打到节点 A 时，完整链路如下：

- **本地优先，未命中则计算路由：** 请求先查本地 `main_cache` 或 `hot_cache`，命中则直接返回（`src/geecache.cpp:145`）。未命中时，通过 `PickPeer(key)` 在哈希环里算出 owner 节点。如果 owner 不是自己，就返回对应的远端 HttpGetter。路由规则本质即：**`key --一致性哈希--> owner 节点`**（`src/http.cpp:108`）。
- **远端 HTTP 取值与防循环转发：** 若 owner 是远端节点 B，A 会发送 HTTP 请求（`GET /_geecache/<group>/<key>`）去取值（`src/http.cpp:378`）。B 收到带有 `X-Geecache-Peer` 头的请求后，只会调用 `GetLocallyOwned(key)` 做本地处理，绝不二次转发给 C（`src/http.cpp:207`）。这确保了**路由只发生一次**，避免请求在节点间来回打转。
- **远端返回与本地回填：** B 返回值给 A 后，A 发起请求的节点会在 `MaybePopulateHotCache` 里做概率性的本地副本回填（`src/geecache.cpp:173` 和 `241`）。

**读请求与失效请求的路由差异：**

- **读路由：** 单点定向（按 key 精准找 owner）。
- **失效路由：** 全量广播（直接广播给所有 peer，见 `src/geecache.cpp:95` 和 `src/http.cpp:405`）。

**一句话总结：** <font color=24b3aa>节点间数据路由是通过 HTTPPool 维护的一致性哈希环完成的：本地 miss 时先根据 key 计算 owner 节点，如果是远端就通过 HTTP 定向访问；远端收到 peer 请求后只做本地处理，不再二次转发。</font>

>   <font color=blue>“路由”</font> 在这里的核心含义，就是决定这个请求或者这个 key 应该交给哪个节点处理。和路由器的‘选下一跳/选目标’这个思想很像，只不过这里选的是业务上负责这个 key 的缓存节点。
>
>   当前节点收到一个读请求后，会先查本地缓存，也就是 main_cache（节点内部的主缓存） 和 hot_cache（缓存远端 owner 返回的数据副本）。如果本地命中就直接返回；如果 miss，就通过 <font color=blue>一致性哈希</font> 计算这个 key 对应的 owner 节点。如果 owner 是当前节点，就由当前节点本地回源加载据；如果 owner 是远端节点，就通过 HTTP 去请求那个节点获取数据。远端节点返回后，当前节点还可以把结果按概率回填到本地 hot_cache，用于减少后续重复的跨节点访问。

## 你是怎么实现远端节点访问的？

这个项目本质上是把节点间通信做成了一套极简的 <font color=blue>HTTP RPC</font>。每个节点的核心是一个兼具“服务端”与“客户端”双重角色的 `HTTPPool`。

**1. 节点的双重身份 (HTTPPool)**

- **作为服务端**：<font color=blue>节点启动时会在本地起 HTTP 服务，对外暴露 `GET`（读取）和 `DELETE`（失效）两个 `/_geecache/` 接口。</font>请求进入网关后，会解析出 Group 和 Key 并在本地执行查询或清理。
- **作为客户端**：在初始化集群 peers 时，会为每个远端节点绑定一个 `HttpGetter`。当本地缓存未命中，节点会找到数据归属的 peer，并通过 `HttpGetter` 主动向远端发起 HTTP 请求。

**2. 核心请求链路**

- **读数据 (GET)**：客户端发起 `GET /_geecache/<group>/<key>`。为了防止请求在节点间无限多跳或死循环，请求头会专门带上 `X-Geecache-Peer: 1`。服务端一看到这个标记，就明确“只查本地，绝不再转发”。
- **失效广播 (DELETE)**：数据更新时，节点会遍历所有 peers 发起 `DELETE` 请求，并通过请求头 `X-Geecache-Version` 传递当前版本号，远端收到后即可执行本地清理与版本推进。

**3. 高效的数据载荷** <font color=blue>通信协议没有使用臃肿的 JSON，而是把 HTTP 当作纯传输层：URL 路径负责资源定位，Header 负责传“防环”和“版本”等控制信令；而真正的响应结果则全部序列化为 Protobuf 二进制流塞进 Body 中返回，既省带宽又提升了解析性能。</font>

------

一句话总结：这就是一套 <font color=blue>以 HTTP 为传输层、Header 传控制信令、**Protobuf 传核心数据** 的 **极简 P2P 缓存通信架构**。</font>

HTTP 通信层我没有从零实现协议栈，而是基于 cpp-httplib 做了封装；我自己实现的重点是节点路由、peer 通信抽象、请求协议设计，以及缓存一致性相关逻辑。

##   一致性哈希为什么要加虚拟节点？

为了 <font color=blue>让节点在哈希环上的分布更均匀</font>，<font color=blue>减轻负载倾斜</font>，同时 <font color=blue>在节点变动时减少大范围重映射</font>。

## 为什么是 50 个虚拟节点？

因为这里的 50 本质上是一个工程上的折中默认值，不是某个严格推导出来的最优解。项目里把它写成了常量 `kDefaultReplicas = 50`，见 src/http.h:22，然后在构建哈希环时传给 `consistenthash::Map`，见 src/http.cpp:63。Map::Add() 会为每个真实节点生成  0..49 这 50 个虚拟节点，见src/consistenthash/consistenthash.cpp:19。

为什么要这么做，可以从两个方向理解。

<font color=blue>第一，虚拟节点太少会导致分布不够均匀。</font>如果每个真实节点在环上只有 1 个点，节点数量一少，key 很容易集中到某几个节点上。加了虚拟节点以后，一个真实节点会以多个点分散在环上，key 的归属会更平滑，负载更均衡。

<font color=blue>第二，虚拟节点也不是越多越好。</font>因为虚拟节点越多：

  - 哈希环越大
  - <font color=blue>构建和维护成本越高</font>
  - <font color=blue>节点变更时需要处理的环元素也越多</font>

所以要在“分布均匀性”和“实现开销”之间找一个平衡。50 在这种教学/中小规模项目里通常是一个比较顺手的默认值：

  - 对几台到十几台节点的小集群来说，节点数 * 50 已经能把环打散得比较均匀
  - 开销又不大，代码和测试都简单

  - 3 个节点时，环上大约有 150 个点
  - 10 个节点时，环上大约有 500 个点

这个量级通常足够避免“环太稀疏”的问题。

不过要说严谨一点：

这个仓库里并没有 benchmark 证明“50 一定最好”。所以面试里最稳的说法不是“50 是最优值”，而是：

“我这里用 50 个虚拟节点，是为了让一致性哈希分布更均匀，同时把实现和维护成本控制在比较低的范围内。<font color=blue>这是一个工程上的经验性默认值，不是仓库里经过专门压测证明的最优参数。</font>”

## hotCache 是怎么做的？

hotCache 在这个项目里，本质上是一个非 owner 节点上的本地热点副本缓存，<font color=blue>用来减少重复跨节点访问</font>。

最核心的理解是：

  - main_cache 存的是“当前节点自己负责的 key”
  - hot_cache 存的是“当前节点不是 owner，但之前从远端 owner 读回来过的值”

代码上，Group 里同时维护了这两层缓存，见 src/geecache.h:132。

<font color=blue>1. hotCache 怎么初始化</font>

在构造 Group 时，会初始化 hot_cache_ 和它的回填参数

* <font color=blue>hot_cache_bytes 默认是主缓存容量的 1/8</font>
* hot_cache_denominator 默认是 8

对应代码在 src/geecache.h:20 和 src/geecache.cpp:21。

也就是说，默认语义大概是：

  - hotCache 容量比主缓存小
  - 不是所有远端读回来的 key 都会回填进去

<font color=blue>2. 读请求时怎么用</font>

本地读请求会先查两层缓存：

```cpp
std::pair<ByteView, bool> Group::LookupCaches(const std::string& key) {
    auto [view, ok] = main_cache_.Get(key);
    if (ok) {
        return {view, true};
    }
    return hot_cache_.Get(key);
}
```

见 src/geecache.cpp:270。

所以顺序是：

    1. <font color=blue>先查 main_cache</font>
    2. <font color=blue>再查 hot_cache</font>
    3. 两层都 miss 才进入加载流程

也就是说，hotCache 是本地缓存命中的第二层兜底。

<font color=blue>3. 什么时候往 hotCache 里放数据</font>

关键点在这里：只有“从远端 peer 成功取回值”之后，才会考虑写入 hot_cache_。

在 src/geecache.cpp:181 的 Load() 里：

```cpp
if (mode == LoadMode::kAllowPeer && peers) {
    auto [peer, ok] = peers->PickPeer(key);
    if (ok) {
        auto [value, peer_err] = GetFromPeer(peer, key);
        if (peer_err.empty()) {
            if (!IsVersionCurrent(key, version)) {
                continue;
            }
            MaybePopulateHotCache(key, value);
            return {std::any(value), ""};
        }
    }
}
```

  这段逻辑的意思是：

  - 当前节点发现这个 key 的 owner 是远端
  - 去远端取值成功
  - **先做 <font color=blue>版本校验</font>，防止并发失效时旧值回填**
  - 然后调用 MaybePopulateHotCache(key, value)

  所以 hotCache 不是本地回源后写进去的，而是远端访问成功后，给非 owner 节点留一份本地副本。

<font color=blue>4. “热点”是怎么判断的</font>

<font color=blue>这里其实没有真正做访问频次统计，也没有 LFU/LRU 热点识别算法。</font>它实现的是一种 <font color=blue>基于 key 哈希的概率性回填</font>。

代码在 src/geecache.cpp:258：

```cpp
void Group::MaybePopulateHotCache(const std::string& key, const ByteView& value) {
    if (hot_cache_denominator_ <= 1) {
        hot_cache_.Add(key, value);
        return;
    }
    size_t bucket = std::hash<std::string>{}(key) %
        static_cast<size_t>(hot_cache_denominator_);
    if (bucket == 0) {
        hot_cache_.Add(key, value);
    }
}
```

它的含义是：

  - 如果 hot_cache_denominator <= 1，那就是每个远端 key 都回填
  - 否则就做一次：
    hash(key) % denominator
  - 只有结果等于 0 的 key，才会进入 hot_cache

所以默认 denominator = 8 时，大致可以理解成：

  - 平均约 1/8 的远端 key 会被选中回填到本地 hotCache

这里有个细节值得你记住：

<font color=blue>它不是“每次请求随机 1/8 概率写入”，而是“按 key 的哈希结果做稳定采样”。</font>同一个 key 的结果是稳定的，不是每次都摇号。

<font color=blue>5. 一致性怎么保证</font>

在 src/geecache.cpp:143 的 ApplyInvalidation()：

```cpp
main_cache_.Remove(key);
hot_cache_.Remove(key);
```

  也就是说：

  - 先删除 mainCache
  - 再删除 hotCache

  再加上前面的版本校验，能避免“远端旧值晚回来后又写进 hotCache”。

## 版本校验具体解决了什么问题？

版本校验具体解决的是：

* 删除缓存或更新数据的同时，另一个并发读请求把旧值重新写回缓存 这个问题。
* 通常叫 <font color=blue>stale backfill（旧值回填）</font>。

最典型的并发场景就是：

    1. 线程 A 发现缓存 miss，开始去远端 peer 或底层数据源加载值
    2. 线程 B 这时更新了数据，并执行 Invalidate(key)
    3. Invalidate 把缓存删掉了，但线程 A 手里还握着“更新前读到的旧值”
    4. 如果线程 A 此时不做校验，直接把这个旧值写回 main_cache 或 hot_cache
    5. 那缓存就被旧值“复活”了

这就是版本校验要拦住的事。

代码里是怎么做的

失效时，先推进这个 key 的版本号，再删除缓存。

见 src/geecache.cpp:143：

```cpp
uint64_t& current = versions_[key];
if (version > current) {
    current = version;
}
main_cache_.Remove(key);
hot_cache_.Remove(key);
```

也就是说，ApplyInvalidation() 不只是删缓存，还会记录：“这个 key 已经至少更新到这个版本了。”

然后读路径在真正回填前，会先记住当前版本，加载完成后再比一次。

远端 peer 路径在 src/geecache.cpp:181：

```cpp
uint64_t version = GetVersion(key);
...
auto [value, peer_err] = GetFromPeer(peer, key);
if (peer_err.empty()) {
    if (!IsVersionCurrent(key, version)) {
        continue;
    }
    MaybePopulateHotCache(key, value);
}
```

本地回源路径在 src/geecache.cpp:236：

```cpp
uint64_t version = GetVersion(key);
auto [bytes, err] = getter_->Get(key);
...
if (!IsVersionCurrent(key, version)) {
	continue;
}
PopulateMainCache(key, value);
```

判断函数很简单，见 src/geecache.cpp:291：

```cpp
bool Group::IsVersionCurrent(const std::string& key, uint64_t version) const {
    return GetVersion(key) == version;
}
```

意思就是：

  - 读开始时看到的是版本 v1
  - 如果加载回来后，这个 key 当前已经不是 v1 了
  - 说明中途发生过失效或更新
  - 那这次加载到的值就不能回填

它保护了哪两类回填

这个项目里它同时保护了两种回填：

  - 远端 peer -> hot_cache
  - 本地数据源 -> main_cache

所以 <font color=blue>不只是主缓存会被保护，热点副本缓存 hotCache 也会被保护。</font>

## 并发安全怎么做的？

  因为它本质上是一个缓存组件，而缓存天然会遇到大量并发读写场景。
  在你这个项目里，至少有 4 类典型并发。

<font color=blue>1. 多线程同时读同一个 key</font>

这是最常见的情况。

比如某个热点 key 突然大量请求进来：

  - 多个线程几乎同时访问 user:1001
  - 如果本地缓存 miss
  - 那就可能同时去远端 peer 或底层数据源加载

如果不处理，会出现：

  - 重复回源
  - 重复跨节点请求
  - 缓存击穿

  所以 <font color=24b3aa>项目里用了 singleflight 合并同一个 key 的并发 miss，保证同一时刻只有一个线程真正去加载，其余线程等结果。</font>

实现见 src/singleflight/singleflight.cpp:6 和 src/geecache.cpp:181。

---------

<font color=blue>2. 多线程并发访问本地缓存</font>

本地缓存本身也是共享数据结构，需要用读写锁保护。

比如：

  - 线程 A 在 Get
  - 线程 B 在 Add
  - 线程 C 在 Remove

如果没有锁保护，LRU 内部结构可能直接被并发破坏。

所以项目里用 ConcurrentCache 对 lru::Cache 做了并发安全封装，底层用的是 std::shared_mutex。实现见 src/cache.h:15。

这解决的是数据结构层面的线程安全问题。

---------

<font color=blue>3. 读请求和失效请求并发</font>

这是这个项目里最值得讲的一类并发。

典型场景是：

  - 线程 A 发现缓存 miss，开始去远端 peer 或底层数据源加载
  - 线程 B 同时更新数据，并调用 Invalidate
  - B 已经把缓存删了
  - A 晚一点才拿着旧值回来

如果不处理，A 可能把旧值重新写回 main_cache 或 hot_cache。

这就是 stale backfill，旧值回填问题。

项目里为此做了版本校验：

  - Invalidate 时推进版本号
  - 加载前记录旧版本
  - 回填前比较版本
  - 版本不一致就丢弃本次结果

实现见 src/geecache.cpp:143、src/geecache.cpp:181、src/geecache.cpp:236。

 解决的是并发一致性问题，不是单纯的线程安全。

## 如果热点 key 很多，这个方案（一致性哈希）会不会有问题？

会有问题，而且这是这个方案很值得面试里主动讲的 tradeoff。

一致性哈希只能解决“数据分布尽量均匀”，不能解决“访问分布极度倾斜”。如果某几个 key 特别热，它们仍然只会落到某一个 owner 节点上，所以 owner 可能成为热点瓶颈，即使环本身分布是均匀的。

这个项目用 hot_cache 缓解了一部分问题：非 owner 从远端拿到值后，可以把它顺手缓存到本地，减少后续跨节点访问，逻辑在 src/geecache.cpp:236。但它只
  是“缓解”，不是彻底解决，因为：

  - 热点第一次还是会打到 owner。
  - hot_cache 只是副本缓存，会带来额外内存复制。
  - 热点 key 一多，hot_cache 也可能被挤爆，造成缓存污染。
  - 失效时仍然要广播所有 peer，热点越多，失效风暴越明显，见 src/geecache.cpp:88。

另外，这里的 hot_cache 不是基于真实访问频率做准入，而是按 hash(key) % denominator 做确定性采样，见 src/geecache.cpp:240。所以如果热点 key 很多，它未必总能把“最值得副本化”的那些 key 留下来。

## 项目中有用到设计模式吗？

在设计项目时其实并没有专门套用设计模式来设计，不过随着功能的拆分，其实有一些设计模式是自然形成的。

<font color=blue>最明显的是策略模式。</font>

项目把可变的行为都抽成了接口，比如：

  - Getter / GetterFunc：把“怎么回源取数据”抽象出来，见 src/geecache.h:26
  - Setter / SetterFunc：把“怎么写回底层存储”抽象出来，见 src/geecache.h:47
  - HashFunc：<font color=24b3aa>一致性哈希支持注入不同哈希函数</font>，见 src/consistenthash/consistenthash.h:12

> 我们这里使用的是 CRC32（循环冗余校验）。
>
> CRC32 原本是用来检测数据传输错误的，但因为它能把任意数据（比如服务器 IP 或缓存 Key）快速变成一个 0 到 42亿（2^32-1）的数字，所以经常被“跨界”拿来做一致性哈希的计算函数。
>
> 它在一致性哈希里非常经典，主要是因为这三个核心优势：
>
> - **天然契合哈希环**：它的输出结果是一个 32 位无符号整数，正好完美对应一致性哈希里那个首尾相接的“环形”寻址空间。
> - **计算速度极快**：它比 MD5 或 SHA 系列等加密型哈希算法轻量得多，在分布式系统里高频计算节点和数据位置时，几乎没有性能负担。
> - **打散效果不错**：虽然是个校验算法，但它能把输入的数据相对均匀地散列在环上，基本能满足早期系统负载均衡的需求。

这本质上都是“把算法/行为做成可替换策略”。

<font color=blue>第二个比较明显的是远程代理模式。</font>

> 远程代理模式（Remote Proxy Pattern）其实非常直白，**就像是你生活里的“海外代购”**。你想买国外的商品，但不想自己买机票飞过去，于是你找了本地的代购。你只管给代购交钱提需求，代购负责跨国沟通和物流，最后把东西交到你手上。
>
> 在软件开发中，它的核心目的就是**让“调用远程服务器上的对象”感觉就像“调用本地对象”一样简单**。
>
> 掌握它只需记住以下 3 个核心点：
>
> - **隐藏网络细节：** 代理对象在本地“伪装”成真正的目标对象。它偷偷把你的方法调用打包，通过网络发给远端，等远端处理完把结果传回来后，再交给你。你根本不需要操心怎么建连接、怎么发 HTTP 或者是 TCP 请求。
> - **位置透明性：** 作为调用方（客户端），你完全感知不到代码是在别的机器上跑的，你的体验就是顺滑地调了一个本地函数。
> - **常见应用场景：** 我们平时常听说的 RPC（远程过程调用）、微服务架构里的各种服务调用（比如 gRPC、Dubbo）、甚至网页前端调用后端 API 的某些封装，本质上都在使用远程代理的思想。

HTTPPool::HttpGetter 这个内部类本质上就是一个远程代理。

它实现了 PeerGetter 接口，但内部不是本地取值，而是帮你发 HTTP 请求到远端 peer，然后把响应解析回来，http.h (line 55) http.cpp (line 378)。

也就是说，对 Group 来说，它只知道自己拿到了一个 PeerGetter：

> 外观模式（Facade Pattern）就像是你家里的**“智能音箱”**。

>
> 在写代码时，它的核心作用就是**把复杂的子系统打包，对外提供一个极简的“傻瓜式”接口**。
>
> 要掌握外观模式，记住这 3 个核心点即可：
>
> - **提供统一入口：** 当你需要调用一堆互相交织的复杂类库时，外观模式会为你建一个“前台接待”。你直接向“前台”提需求，由它去和后面复杂的子系统打交道，你完全不用去搞懂那些底层代码是怎么运转的。
> - **降低代码耦合：** 你的客户端代码只和“外观类”发生联系。哪天底层的子系统代码大换血了，只要“外观类”对外承诺的接口不变，你的业务逻辑就一行都不用改。
> - **不限制底层访问：** 外观模式只是为你提供了一条捷径，它并没有把底层系统锁死。如果你有非常个性化的高级需求，随时可以绕开这个“前台”，直接去调用底层的细分组件。
>
> 总而言之，当你面对一个像蜘蛛网一样错综复杂的遗留系统，或者要引入一个功能庞杂的第三方库时，自己写一个“外观类”把最常用的功能简单封装一下，是保持代码整洁和理智的最佳手段。

Group 本身就是整个缓存组件对外的统一入口，外部只需要关心：

  - Get
  - Set
  - Invalidate
  - RegisterPeers

见 src/geecache.h:91。

但在 Group 内部其实隐藏了很多复杂度：

  - main_cache / hot_cache
  - singleflight
  - peer 路由
  - 版本校验
  - 失效广播

所以<font color=blue>Group 很像一个外观对象，把复杂子系统封装在后面</font>。

## Group 是什么？

**<font color=red>1. 核心定位：缓存系统的“业务总管”</font>** Group 本质上是一个**缓存命名空间**，也是业务层访问缓存（Get/Set/Invalidate）的核心入口。<font color=blue>在项目中，某一类业务数据（比如“scores”）的整套读写策略，都会独立封装在一个 Group 里，避免不同业务的数据和配置（如容量、TTL、回源逻辑）混在一起。</font>

**<font color=red>2. 职责边界：Group 管缓存，HTTPPool 管通信</font>**

- **<font color=blue>Group 负责缓存语义</font>**：它内部封装了本地主缓存、热点缓存、回源逻辑（getter）、请求合并（singleflight）以及版本校验。
- **<font color=blue>HTTPPool 负责节点协作</font>**：负责一致性哈希选节点、HTTP 收发通信和失效广播。 两者配合时，业务层面只跟 Group 打交道，底层的复杂链路（查本地、找远端、触发回源）全被 Group 屏蔽了。

**<font color=red>3. 分布式设定：同名即同域，Owner 动态分配</font>** 参与分布式协作的节点，必须在各自进程里创建**名字相同**的 Group 实例。名字相同代表它们逻辑上属于同一个缓存域，这样节点间的请求才知道该交给本地的哪个对象处理（名字不同会直接报错 404）。但是，具体某个 Key 最终归谁管（Owner），是通过一致性哈希动态算出来的，每个节点的本地状态和缓存内容并不完全一样。

**<font color=red>4. 关键底线：同名 Group 的回源语义必须一致</font>** 不同节点上同名 Group 的底层数据抓取逻辑（getter）可以底层实现不同，但**业务语义必须严格一致**。因为正常情况下，数据由 Owner 节点的 getter 提供；一旦网络抖动或远端不可用，请求会降级使用当前节点的本地 getter。如果语义不一致，就会导致同一个 Key 在降级或扩缩容时，读出毫无关联的结果，造成业务语义漂移。

## HTTPPool 是什么？

HTTPPool 可以理解成分布式节点间通信和路由的核心组件。它一方面负责维护 peer 列表、构建一致性哈希环、根据 key 选择 owner 节点；另一方面也负责启动本地 HTTP 服务、接收其他节点的请求，并通过内部的 HttpGetter 访问远端 peer。另外，健康检查和失效广播也是通过 HTTPPool 完成的。<font color=blue>简单来说，Group 负责缓存语义，HTTPPool 负责节点协作。</font>

HTTPPool 是由使用方主动创建的，不是框架自动生成的。调用方会先根据当前节点地址实例化一个 HTTPPool，再把 peer 列表传进去，接着把它和本地的 Group 互相注册，最后启动 HTTP 服务。也就是说，组件负责提供节点通信能力，但具体怎么初始化和接入，是由上层使用方决定的。

>
> 项目摒弃了传统 C++ 的手动管理，大量采用 C++11 及以后的特性来保证内存和异常安全：
>
> - **智能指针明确所有权**：通过 `std::shared_ptr` 共享 Group、Getter 等核心对象的生命周期，避免手动释放；使用 `std::unique_ptr` 独占一致性哈希环和 LRU 淘汰组件的资源，杜绝内存泄漏。
> - **RAII 风格的锁管理**：项目里的锁基本都交由 `std::lock_guard`、`std::unique_lock` 和 `std::shared_lock` 自动管理。获取即初始化，保证了发生异常时锁也能安全释放。
> - **移动语义减少拷贝**：大量使用 `std::move` 进行资源所有权的转移（如字符串处理、闭包参数传递），避免了不必要的深拷贝，性能更贴合现代 C++ 标准。
>
> ### 2. 高效且场景化的并发控制
>
> 作为一个缓存系统，高并发下的数据一致性是核心，项目在不同场景使用了最适合的并发组件：
>
> - **读写锁分离 (`std::shared_mutex`)**：本地缓存使用了读写锁。读操作并发，写操作独占，完美契合缓存系统“读多写少”的业务特征。
> - **并发请求合并 (`std::mutex` + `std::condition_variable`)**：在 Singleflight 机制中，利用互斥锁和条件变量让后续相同的 Key 请求等待首个线程加载完成，从而合并请求，有效防止缓存击穿。
> - **原子变量无锁递增 (`std::atomic`)**：对于全局版本时钟、状态标志位等高频访问的共享状态，使用原子变量保证线程安全和递增性能。
>
> ### 3. 灵活的策略抽象与泛型
>
> 项目在接口设计上非常注重解耦和可扩展性，大量使用了函数式编程的思想：

> - **策略注入 (`std::function` + Lambda)**：将核心行为抽象为可替换的策略。比如通过 `std::function` 定义 `GetterFunc` 和 `HashFunc`，并配合 Lambda 表达式在调用处直接内联回调逻辑，让接口极其灵活。
> - **类型擦除 (`std::any`)**：在 Singleflight 的结果传递中，利用 `std::any` 来承载底层返回的数据。这让并发合并组件不再强耦合于具体的缓存数据类型，提升了代码的通用性。

## 分布式节点间的并发怎么处理？

布式节点间的并发处理主要不是靠分布式锁，而是靠<font color=red>“owner 路由 + 请求合并 + 失效广播 + 版本校验”</font>这套组合。

先把两层问题分开：

<font color=blue>本地单机并发，会用锁。</font>

  - 本地缓存 ConcurrentCache 用 std::shared_mutex 保护，见 src/cache.h:15。
  - versions_、groups_、peers_ 这些共享状态也分别用 shared_mutex / mutex 保护，见 src/geecache.h:138 和 src/
    geecache.h:141。

<font color=blue>分布式节点间并发，不靠锁，而是这样处理：</font>

    1. 同一个 key 先路由到同一个 owner 节点
       - 本地 miss 后先用一致性哈希选 owner。
       - 如果 owner 是远端，就通过 peer 请求把流量汇聚到那个节点。
       - 代码在 src/http.cpp:108 和 src/geecache.cpp:181。
    2. owner 节点对同一个 key 做请求合并
       - 即使 A、C 两个节点同时请求同一个 key，最后都会落到 owner B。
       - B 上的 singleflight 会把同一个 key 的并发加载合并成一次真正回源。（<font color=24b3aa>当然这里会对调用方的保存做加锁处理。</font>）
       - 代码在 src/singleflight/singleflight.cpp:6 和 src/geecache.cpp:181。
    3. peer 请求到达 owner 后不再继续转发
       - 请求头里带 X-Geecache-Peer，服务端看到后直接走 GetLocallyOwned。
       - 这样不会出现 A 找 B，B 又继续转发的多跳并发问题。
       - 代码在 src/http.cpp:207。
    4. 更新时用失效广播清各节点副本
       - 写成功后调用 Invalidate，向所有 peer 发 HTTP DELETE。
       - 对端收到后执行 ApplyInvalidation，删除本地 main_cache 和 hot_cache。
       - 代码在 src/geecache.cpp:95 和 src/http.cpp:190。
    5. 版本校验解决“旧值晚归”
       - 这是分布式并发里最关键的一点。
       - 场景是：A 正在从 B 读旧值，C 同时更新并广播失效。如果没有版本校验，A 晚一点收到旧值后可能又把旧值写回缓存。
       - 现在的做法是：失效时先推进版本，再删缓存；读取回填前先比较版本，不一致就丢弃。
       - 代码在 src/geecache.cpp:143、src/geecache.cpp:197、src/geecache.cpp:245。

总结：单机内共享数据结构是靠锁保护的，但分布式节点间并发不是靠分布式锁。这个项目的做法是先通过一致性哈希把同一个 key 汇聚到同一个 owner，再在 owner 上用 singleflight 合并并发加载；写路径通过失效广播清除各节点副本，并用版本校验防止并发场景下旧值重新回填缓存。”

所以它解决的是最终一致性下的分布式并发问题，不是通过分布式锁去做严格强一致。

## 广播失效失败了怎么办？

MySQL 更新成功后，我们走 Cache Aside，先删当前节点缓存，再通过 HTTP 向其他节点广播失效。当前实现里，这个广播是 <font color=blue>best effort（最终一致性）</font> 的：会尽量把所有 peer 都通知到，但 <font color=blue>如果有节点失效失败，接口会把错误返回出来</font>，而不是默默吞掉。

```cpp
td::string Group::Invalidate(const std::string& key) {
    if (key.empty()) {
        return "key is required";
    }

    uint64_t version = NextVersion();
    std::string err = ApplyInvalidation(key, version);
    if (!err.empty()) {
        return err;
    }

    std::shared_ptr<PeerPicker> peers;
    {
        std::lock_guard<std::mutex> lock(peers_mu_);
        peers = peers_;
    }
    auto manager = std::dynamic_pointer_cast<PeerManager>(peers);
    if (!manager) {
        return "";
    }

    geecachepb::Request req;
    req.set_group(name_);
    req.set_key(key);
    req.set_version(version);

    std::vector<std::string> errors;
    for (const auto& peer : manager->GetAllPeers()) {
        std::string peer_err = peer->Invalidate(req);
        if (!peer_err.empty()) {
            errors.push_back(peer_err);
        }
    }
    if (errors.empty()) {
        return "";
    }

    std::ostringstream oss;
    oss << "failed to invalidate " << errors.size() << " peer(s): ";
    for (size_t i = 0; i < errors.size(); ++i) {
        if (i > 0) {
            oss << "; ";
        }
        oss << errors[i];
    }
    return oss.str();
}
```

发送方会遍历所有 peer，逐个调用失效接口尝试通知它们

  - 但系统不保证“所有 peer 最终都成功失效”。如果网络失败、超时、对端不可达，或者返回非 200，发送方就把这个 peer 记为失败
  - <font color=blue>对失败 peer，当前实现只返回错误，不会自动重试，也没有 MQ / Outbox 这类补偿机制。</font>

  - 如果发送方收到了 200，那说明接收方已经执行了本地 ApplyInvalidation，因为 HTTP DELETE 处理逻辑是先执行失效，再返回 200
  - 如果发送方拿到的是超时、连接失败、非 200，那就只能认为“这个节点的失效没有被确认成功”。工程上应当把它视为失败处理。
  - <font color=24b3aa>严格讲，网络场景下还存在一种经典歧义：接收方可能已经执行了，但响应在路上丢了。当前代码没有幂等确认或补偿机制，所以发送方无法区分这种情况。</font>

> 接受失效广播的 peer 可能失败的情况：
>
> * 请求根本没送到。比如节点宕机、HTTP 服务没起来、端口不通、网络故障、连接超时、读超时。发送方会在 src/http.cpp:415 这里
>   拿到空结果，然后返回 http invalidation failed ...。
>   * 请求到了，但接收端返回了非 200。发送方会把任何非 200 都视为失败，见 src/http.cpp:421。
>   * 非 200 里当前代码明确有几类：
>     - 400：路径不对，或者 URL 里缺少 key，见 src/http.cpp:150 和 src/http.cpp:164。
>     - 404：接收节点上没有这个 group，通常是节点配置不一致、没注册 group、或者服务状态异常，见 src/http.cpp:181。
>     - 500：ApplyInvalidation 返回错误，当前实现里基本就是前面说的空 key，见 src/http.cpp:196。

失败的那个节点，短时间内仍然可能保留旧数据。这里我用了两层兜底：

    1. <font color=blue>版本校验</font>
       针对“删除和加载并发”场景，避免旧值回填。
       也就是某个节点已经收到了失效通知、版本已经前进，那么它手里晚回来的旧值就不能再写回缓存。
    2. <font color=blue>TTL 兜底</font>
       如果某个节点因为广播失败，根本没收到这次失效，那版本校验帮不上忙，因为它连“版本变了”都不知道。这时候靠 TTL 保证旧缓
       存不会无限期存在，过期后会自动淘汰并重新回源。

<font color=red>如果面试官继续追问**“那更好的线上方案呢”**，我会补一句：</font>

如果业务对一致性要求更高，下一步应该加<font color=blue>“异步重试机制”</font>，比如：

  - 失败节点写入重试队列
  - 用 MQ / Outbox 保证失效消息最终投递
  - 配合告警和重试次数控制

  这样就能把当前的 best effort + TTL，升级成“失败可恢复”的一致性方案。

> MQ 是消息队列，Outbox 是一种可靠投递模式。两者经常一起用，但不是一回事。
>
> ### 先说 MQ：
>
> - MQ = Message Queue / Message Broker，比如 Kafka、RabbitMQ、RocketMQ。
> - 它的作用是把“我要通知别的节点失效缓存”这件事，改成“我先发一条消息到中间件，由中间件负责存储、转发、重试、削峰”。
> - 这样发送方就不用同步遍历每个 peer 逐个 HTTP 调了，节点之间也解耦。
>
> 放到你的场景里，思路就是：
>
> 1. MySQL 更新成功。
> 2. 发送一条“key X 已更新，需要失效”的消息到 MQ。
> 3. 各个缓存节点订阅这个消息。
> 4. 收到消息后执行本地 invalidate(key, version)。
>
> 它比现在的同步 HTTP fan-out 更稳的点在于：
>
> - 消息可以持久化，不是“发一枪就算了”。
> - 消费失败可以重试。
> - 发送方不用等所有节点在线。
> - 节点短暂宕机后恢复，也可能继续消费积压消息。
>
> 但单独只有 MQ 还不够，因为还存在一个经典问题：双写不一致。
>
> 比如：
>
> 1. 你先更新了 MySQL，成功。
> 2. 然后准备发 MQ。
> 3. 结果进程在发 MQ 之前崩了。
>
> 那这次变更就已经写进数据库了，但失效消息永远没发出去。这就是“数据库写成功，消息没发出去”的双写问题。
>
> 这时候就轮到 Outbox。
>
> ### Outbox 是什么：
>
> - Outbox 不是中间件，而是一种模式。
> - 核心做法是：把“业务数据变更”和“待发送事件”写进同一个数据库事务里。
>
> 典型做法是多一张表，比如 outbox_events：
>
> - id
> - event_type
> - aggregate_key
> - payload
> - status
> - created_at
>
> 写路径变成这样：
>
> 1. 开启 MySQL 事务。
> 2. 更新业务表。
> 3. 同一个事务里插入一条 outbox 事件，比如“用户123已更新，缓存 key=user:123，version=456”。
> 4. 提交事务。
>
> 这样只要事务提交成功，就能保证两件事同时成立：
>
> - 业务数据真的更新了。
> - “这次更新对应的一条待投递事件”也一定落库了。
>
> 之后再由一个后台 worker 做这件事：
>
> 1. 扫描 outbox_events 里未发送的记录。
> 2. 发到 MQ，或者直接调用失效逻辑。
> 3. 成功后把这条 outbox 记录标记为已发送。
> 4. 失败则稍后重试。
>
> 所以更准确地说：
>
> - MQ 解决的是“消息如何可靠传输、缓冲、消费”。
> - Outbox 解决的是“数据库更新和消息发送之间的双写一致性问题”。
>
> 在工程上，最常见的可靠方案其实不是 “MQ 或 Outbox” 二选一，而是：
>
> **MySQL事务 + Outbox表 + 后台投递器 + MQ + 消费端幂等**
>
> 放到你的缓存失效场景，一条完整链路通常会是：
>
> 1. 更新 MySQL。
> 2. 同事务写入 outbox 事件：invalidate key=Tom version=123。
> 3. 后台任务把 outbox 事件投递到 MQ。
> 4. 各缓存节点消费消息并执行本地失效。
> 5. 如果消费重复了，也没关系，因为失效本来就应当是幂等的；你现在还有 version，更适合做幂等和乱序保护。
>
> 你可以把它和现在的方案对比着理解：
>
> ### 现在的方案：
>
> - 写 MySQL 成功后，立刻同步 HTTP 广播给所有 peer
> - 某个 peer 调用失败就失败了
> - 没有持久化补偿
>
> ### MQ 方案：
>
> - 不直接逐个 peer 打 HTTP
> - 先把事件发到消息系统
> - 由各节点异步消费
>
> ### Outbox 方案：
>
> - 先确保“这条失效事件不会丢”
> - 再异步慢慢投递
> - 即使服务中途崩了，重启后也还能补发

## 某个节点宕机了，会不会出现数据不一致的情况？

会，但不是 “只要有节点宕机就一定不一致”，而是要看它宕在什么时刻。

  - <font color=blue>如果说的是“宕机的那个节点自己”，它不会继续带着旧缓存服务，因为这个项目的缓存是进程内内存；</font>进程一挂，main_cache_ 和 hot_cache_ 就一起没了。节点恢复后更像冷启动，后续读请求会 miss 再加载，而不是继续命中过去那份旧缓存，见 src/
    geecache.cpp:144 和 src/cache.h:32。
  - <font color=blue>但从“整个集群”的角度看，确实可能出现不一致。</font><font color=24b3aa>典型场景是：底层数据已经更新，但负责发失效通知的节点在广播完成前宕机了。</font>当前实现里，Invalidate 只是本地先失效，然后同步遍历 peer 发 HTTP 删除；失败只返回错误，没有重试、没有持久化补偿。这时没收到通知的存活节点，仍可能继续命中自己本地的旧 mainCache 或 hotCache。
  - <font color=blue>版本校验只能防“已看到新版本的节点”把旧值回填回来，解决不了“根本没收到失效通知”的节点；</font>这类脏缓存只能靠 TTL 或更可靠的投递机制来收敛。并且 TTL 默认是关闭的，只有显式配置后才生效。

  一句话回答就是：“单个节点宕机不会让它自己继续拿旧缓存服务，但如果它在更新后、广播失效完成前宕机，就可能让其他还活着的节点暂时持有旧缓存；当前方案保证的是最终一致性，不是强一致。”

## 你的一致性哈希是怎么实现的？

这个项目里的一致性哈希实现比较典型，核心在 src/consistenthash/consistenthash.h 和 src/consistenthash/consistenthash.cpp。

  它的做法是把“真实节点”扩展成很多“虚拟节点”放到一个哈希环上。Map 内部维护的是一个 `std::map<uint32_t, std::string> ring_`，key 是虚拟节点的哈希值，value 是这个虚拟节点对应的真实节点地址。添加节点时，会对每个节点生成 replicas 个虚拟节点，格式是 `to_string(i) + key`，然后对这个字符串做哈希，默认用 crc32（ crc32 返回的是 uint32_t，最大值是： 4294967295），最后放进有序 map 里，见 src/consistenthash/consistenthash.cpp:17 和 src/consistenthash/consistenthash.cpp:21。

查找时，对业务 key 也做一次同样的哈希，然后在 `ring_` 里用 `lower_bound` 找“第一个大于等于这个哈希值的虚拟节点”；如果已经走到末尾，就回到 `begin()`，等价于环形查找。找到的虚拟节点映射到哪个真实节点，这个 key 就归哪个节点，见 src/consistenthash/consistenthash.cpp:30。

  这样实现有三个直接效果：

  - 节点增减时，只有环上相邻区间的一小部分 key 会迁移，不会像普通取模那样大面积重分布。
  - 用虚拟节点可以把数据分布打得更均匀，减少真实节点数量少时的倾斜。
  - 查**找复杂度是 O(log N)，因为底层是有序 std::map。**

  在这个项目里，HTTPPool::Set 会创建这个哈希环，默认副本数是 50，见 src/http.h:20 和 src/http.cpp:53。之后 PickPeer(key) 就是先 peers_->Get(key) 算出 owner，再决定要不要把请求转给远端 peer，见 src/http.cpp:98。

## 为什么不把所有远端数据都回填到本地 hotcache？

> 为什么是概率性回填？为什么不直接全量副本？
>
> 如果热点 key 很多，这个方案会不会有问题？

<font color=blue>这几个个问题本质上都在问一个 tradeoff：hotCache 到底是**“局部加速器”**，还是 <font color=blue>“全量副本”</font>。在你这个项目里，它被设计成前者，不是后者。</font>

先对应一下当前实现：远端读成功后，不是每次都回填本地 hotCache，而是按 key 的哈希做一次固定采样。默认分母是 8，也就是大约只有 1/8 的 key 会被放进 hotCache。<font color=blue>严格说这更像“哈希采样回填”，只是效果上相当于概率性回填。</font>

<font color=red>为什么不把所有远端数据都回填到本地：</font>

  - <font color=blue>因为那会把“分片缓存”慢慢做成“每个节点都在攒全量副本”。</font>节点越多，内存放大越严重。原本每个 key 只需要主要驻留在 owner 节
    点，现在可能在很多非 owner 节点也各留一份。
  - <font color=blue>会污染本地缓存。</font>很多远端 key 只是偶发访问一次，如果全部回填，会把真正值得留在本地的热点数据挤掉。
  - <font color=blue>会扩大一致性面。</font>副本越多，失效广播的命中面越大；一旦广播失败，脏副本也会更多，维护成本更高。
  - 会削弱一致性哈希分片的意义。你已经有 owner 节点负责主要存储了，hotCache 只该解决“跨节点热点读太贵”，不该变成“每个节点
    都存一遍”。

<font color=red>为什么是概率性回填：</font>

  - 它的目标不是绝对命中率最大化，而是用很小的内存代价，换掉一部分重复跨节点流量。
  - 这种做法实现很轻。不需要维护访问计数、热度排行榜、异步晋升逻辑，也不需要额外锁和统计结构。
  - <font color=blue>它能天然限制副本膨胀。你可以把它理解成对远端 key 做一个固定比例的本地抽样缓存。</font>
  - 当前实现还是“按 key 稳定采样”，不是每次访问都随机一次。同一个 key 要么总是能进 hotCache，要么总是不会进，这样行为更稳
    定，也更容易推断容量。

<font color=red>如果热点 key 很多，这个方案会不会有问题：</font>

会，有两个主要问题。

  - <font color=blue>第一，固定 1/N 采样不是自适应的。</font>如果热点非常多，真正该回填的 key 可能只有一部分被采到，剩下的热门 key 还是会频繁跨节点访问。
  - <font color=blue>第二，如果热点集合本身很大，hotCache 容量不够时会抖动，刚回填进来的热门数据可能很快又被别的热门 key 挤掉。</font>
  - 还有一个更隐蔽的问题：<font color=blue>如果某些节点对同一批远端 key 的访问特别集中，而采样比例又固定，就可能出现“这批最热 key 里偏偏很
    多没被采到”，导致 owner 节点仍然承压。</font>

<font color=red>所以这个方案适合什么场景：</font>

  - 远端访问里存在重复读，但不是所有远端 key 都值得本地副本。
  - 你想用很低复杂度，先拿到一部分命中率收益。
  - 你能接受它不是最优，只是一个工程上性价比高的折中。

<font color=red>为什么不用 LFU/LRU 热点晋升</font>

> 核心原因有 4 个。
>
> ### 1. 这里要控制的是“副本扩散”，不是让单机命中率最优
>
> `hotCache` 的本质是**远端数据的本地副本**。
>
> 如果改成 LFU / LRU 晋升，只要某个 key 在某个节点上访问变热，这个节点就会把它收进本地；而当多个节点都这样做时，同一个远端 key 就会在很多节点上扩散。
>
> 这样虽然读性能可能更好，但副作用也很明显：
>
> - 内存副本数上升
> - 失效广播命中面变大
> - 广播失败时脏副本更多
>
> 相比之下，**哈希采样**的好处是：副本比例天然有上限，整体成本更可控。
>
> ------
>
> ### 2. LFU / LRU 用来做“准入”比做“淘汰”复杂得多
>
> LRU 用来做淘汰很简单，因为对象已经在缓存里了。
>
> 但“是否晋升到 `hotCache`”首先要回答一个问题：**用什么热度信号来判断这个 key 足够热？**
>
> - 如果用 **LRU** 做晋升，通常需要先有一个 probation 区、候选区，或者至少要记录“这个 key 最近是否再次被访问过”。
> - 如果用 **LFU** 做晋升，就需要维护访问计数、衰减窗口和老化策略；否则历史高频 key 会长期霸榜。
>
> 这些机制都比当前的 **O(1) 哈希采样**复杂得多，也会带来更多锁竞争和元数据开销。
>
> ------
>
> ### 3. 纯 LRU / LFU 在这个场景下不一定稳定
>
> #### LRU 的问题
>
> LRU 太容易受到“最近一次访问”的影响。
>  一次短暂突发，或者一次扫描流量，都可能把一批实际上不值得长期保留副本的数据推进来。
>
> #### LFU 的问题
>
> LFU 太容易被“历史频率”绑住。
>  昨天很热、今天已经冷掉的 key，如果没有做衰减，仍然可能长期占着 `hotCache`。
>
> 而这里的 `hotCache` 目标，并不是做**全局最优的热点识别**，而是以较低成本削掉一部分重复的跨节点读。
>
> 对于这个目标来说，**简单、稳定、可控**，比“理论上更聪明”更重要。
>
> ------
>

## 节点之间为什么选 HTTP？为什么返回体还要用 Protobuf？

这个项目里节点间选 HTTP，主要是工程实现成本和可验证性之间的平衡；返回体用 Protobuf，是在 HTTP 之上补一个明确、高效的二进制协议层。

<font color=red>为什么选 HTTP</font>

第一，HTTP 最容易落地。这个项目直接用 vendored 的 cpp-httplib，服务端和客户端都很轻，HTTPPool 同时承担服务监听和 peer 调用，代码集中在 src/
  http.cpp 和 src/http.h。

第二，调试简单。路径就是 /_geecache/<group>/<key>，健康检查就是 /_geecache/healthz，出问题时抓包、curl、日志都直观，见 src/http.cpp:135。

第三，它符合这个项目的目标。这里重点是验证分布式缓存机制，比如 peer 路由、singleflight、防旧值回填、失效广播，不是做极致 RPC 框架，所以 HTTP 足
  够了。

第四，跨语言和后续扩展友好。如果以后业务节点不全是 C++，HTTP 比自定义 TCP 协议更容易接入。

  <font color=red>为什么不直接返回裸字符串</font>

因为节点间通信不只是“传一个 value”这么简单，请求和响应都在演进。现在请求里已经有 group、key、version，见 proto/geecachepb.proto。如果一直靠 URL 和纯文本 body，字段一多就会越来越脆弱。

<font color=red>为什么返回体用 Protobuf</font>

第一，类型明确。响应体是 geecachepb::Response，而不是“默认大家都知道 body 里是什么格式”，见 src/http.cpp:196。

第二，二进制更适合缓存值。这里 value 是 bytes，不要求一定是 UTF-8 文本，Protobuf 对二进制 payload 更自然。

第三，扩展性更好。以后要给响应加版本号、源节点、压缩标记、错误细节，只要往 .proto 里加字段即可，兼容性比手搓文本协议强。
  第四，序列化/反序列化开销低，协议紧凑，比分散地拼 JSON 或手写解析更稳。

这个项目里的实际分工

可以把它理解成两层：

  - HTTP 负责“传输”和“路由”
  - Protobuf 负责“消息格式”

比如：

  - 请求的 group 和 key 目前仍体现在 URL 路径里
  - 失效版本号走 HTTP header：X-Geecache-Version
  - 真正的返回值 body 用 Protobuf 编码，见 src/http.cpp:201

所以它不是“HTTP 和 Protobuf 二选一”，而是“HTTP 承载 Protobuf”。

## 一致性的级别有哪些？

可以把“一致性级别”分成两个视角来看，不然面试里很容易混。

**<font color=blue>第一种视角：分布式系统里常说的强弱一致性级别</font>**

这个最常用，也最适合你现在这个项目。

常见可以这样理解：

**1. 强一致性 Strong Consistency**
写成功之后，后续任意节点读到的都必须是最新值。
用户感觉上就像系统里永远只有一份最新数据。

**2. 线性一致性 Linearizability**
可以理解成更严格、形式化定义下的强一致。
所有操作看起来像按真实时间顺序一个一个立即生效。
很多场景下，面试里把它和强一致放在一起讲是可以的。

**3. 顺序一致性 Sequential Consistency**
所有节点看到的操作顺序一致，但这个顺序不一定严格等于真实时间顺序。
比线性一致性弱一点。

**4. 因果一致性 Causal Consistency**
如果两个操作存在因果关系，比如“先写后读”，那大家必须看到一致顺序；
如果没有因果关系，不同节点看到顺序可以不同。

**5. 会话一致性 Session Consistency**
站在单个客户端会话的视角保证一致性。
比如你自己刚写完，接下来在自己的会话里至少能读到自己写的内容。
常见形式包括：

- Read Your Writes
- Monotonic Reads
- Monotonic Writes

**6. 最终一致性 Eventual Consistency**
短时间内允许不同节点看到不同值，但如果后续不再发生新写入，最终大家会收敛到同一个值。
很多缓存系统、DNS、NoSQL 系统都属于这一类。

------

**<font color=blue>第二种视角：数据库隔离级别里的“一致性相关语义”</font>**

这个和你现在说的分布式缓存一致性不是一回事，但面试官有时会混着问。

比如：

- 读未提交
- 读已提交
- 可重复读
- 串行化

这些更偏事务隔离级别，不是你这里“节点间缓存一致性”的重点。
如果面试官问的是你这个项目，通常他说的“一致性级别”更可能是前面那组：强一致、最终一致这些。

## 更新 peer 后旧的热点副本怎么办？

当前实现里，更新 peer 列表时，旧的 main_cache 和 hot_cache 都不会自动处理。

<font color=blue>HTTPPool::Set 只会重建一致性哈希环、更新 http_getters_ 和健康状态，不会碰 Group 里的缓存。</font>

这意味着会有几个后果：

  - 旧的 hot_cache 会继续留着，直到被 Invalidate、TTL 或 LRU 淘汰。
  - 旧的 main_cache 也会继续留着，即使这个节点在新的哈希环下已经不再是 owner。
  - 更关键的是，请求打到这个节点时，如果本地缓存命中了，它会直接返回，不会先按新环重新判断 owner。也就是说，拓扑变更后，旧缓存可能绕过新的 owner 路由继续被读到。

所以从“当前代码行为”来说，答案是：

  - 不会迁移
  - 不会清空
  - 不会按新 owner 自动重分布
  - 只能依赖失效广播、TTL、LRU 逐步把旧数据清掉

 其实是一个可以主动讲的设计缺口。

<font color=blue>如果面试官追问“应该怎么做”，比较合理的回答有两档：</font>

  - 简单保守方案：peer 列表变化时直接清空所有 hot_cache，必要时连 main_cache 一起清空。优点是简单安全，缺点是会产生冷启动。
  - 更细方案：引入拓扑版本 epoch。peer 变更时增加 epoch，缓存项带上写入时的 epoch；读取时如果发现缓存项 epoch 过期，就丢弃并按新 ring 重新加载。hot_cache 可以全部清掉，main_cache 可以按 owner 是否仍归本节点做惰性淘汰。

## 你这个项目是实现了什么级别的一致性？

> 这个项目实现的是最终一致性，不是强一致性。它采用 Cache Aside 写路径，更新后通过本地失效和远端失效广播来让缓存逐步收敛；同时又通过版本校验防止删除和加载并发时的旧值回填，再用 TTL 作为兜底机制限制旧数据停留时间。所以整体上它保证的是工程意义上的最终一致性，而不是分布式数据库那种强一致语义。

这个项目实现的是 **最终一致性**，不是 **强一致性**。

更准确一点说，它是一套：

**<font color=blue>基于 Cache Aside、失效广播、版本校验和 TTL 兜底的最终一致性方案。</font>**

<font color=red>为什么不是强一致？</font>

因为这个项目的写路径是：

1. 先更新后端数据源
2. 再失效本地缓存
3. 再广播失效给其他节点

这本质上是 **Cache Aside**。

这种方案的特点是 **实现简单、性能好**，但在广播延迟、失败，或者某些节点还没来得及重新加载新值的时候，**短时间内可能读到旧值**。

另外，当前实现里：

- 没有分布式事务
- 没有 quorum
- 没有共识协议
- 失效广播也是 best-effort

所以它不可能达到线性一致性或强一致。

<font color=red>为什么又不能说“完全没一致性”？</font>

因为它也做了两层很关键的保护。

**第一层：版本校验，防旧值回填。**

如果某个线程正在加载旧值，另一个线程这时把 key 更新并失效了，旧值不会被重新写回缓存。这个机制解决的是“删除缓存和回源加载并发时的 stale backfill 问题”。

**第二层：TTL 兜底。**

即使某些失效广播漏了，旧数据也不会永久存在，会在过期后被淘汰。

所以虽然它不是强一致，但也不是完全放任不管，而是在工程上尽量把旧值窗口收窄。

## 你这里 HTTP 通信是自己写的还是用了第三方库？

底层 HTTP client/server 我用的是第三方 cpp-httplib，但 geecache 的节点通信协议、失效广播、健康检查和 peer 管理是我自己在它上面封装的。

<font color=red>为什么选 cpp-httplib？</font>

cpp-httplib 是一个轻量级、header-only 的 C++ HTTP 库。它给你现成的 HTTP Server 和 HTTP Client 能力，所以你不用自己从 socket、连接管理、HTTP 报文解析这些底层细节开始写。

  - 客户端只需要向其他节点发 GET 和 DELETE。
  - 它接入成本很低，不需要引入更重的网络框架，也不用自己处理 HTTP 编解码。
  - 对这个 geecache 项目来说，重点在缓存语义、一致性哈希、singleflight、失效广播这些逻辑，不在底层网络栈本身，所以用一个现成轻量 HTTP 库更合适。

<font color=red>你知道 muduo 吗？</font>

muduo 是一个比较经典的 C++ 网络库，偏 Linux 服务端开发，核心定位是高性能、基于 Reactor 模型的 TCP 网络编程库，作者是陈硕。

它的几个典型特点是：

  - 基于 epoll
  - one loop per thread 的线程模型
  - 核心抽象比较清晰，比如 EventLoop、Channel、Poller、TcpConnection、TcpServer
  - 强调非阻塞 IO 和事件驱动

  - muduo 是更底层的网络框架，偏 TCP/事件循环基础设施

## 版本号和锁的区别是什么？

区别可以概括成一句话：

* <font color=blue>**锁** 解决的是“同一时刻谁能改这块内存”</font>
* <font color=blue>**版本号** 解决的是“我现在拿到的值是不是已经过时了”。</font>

放到你这个项目里看，会更清楚。

  - <font color=blue>**锁** 是 <u>并发控制手段</u>，主要保护本地内存结构不被并发读写搞坏。</font>比如 ConcurrentCache 里对 Add/Get/Remove 都用了读写锁，防止多
    个线程同时改 LRU 状态。versions_ 这张版本表本身也要靠锁保护。
  - <font color=blue>**版本号** 是 <u>一致性控制手段</u>，它不负责互斥，而是负责判断“这次加载回来的值还能不能写回缓存”。</font>失效时会先把 key 的版本推进；加载时先记下旧版本，等远端返回或本地回源完成后再检查版本是否还是原来的，如果变了，就丢弃这次结果。

最核心的差别是这几个点：

  - **锁** 是阻塞式的。拿不到锁就要等，目的是串行化访问。
  - **版本号** 是校验式的。它不阻塞别人，只在最后判断结果是否过期。
  - **锁** 主要作用在单机进程内。
  - **版本号** 可以跨线程、跨请求，甚至跨节点传播，所以更适合处理缓存失效这种“时间上拉长”的并发问题。

为什么这里不能只靠锁？

* 因为你这个竞态不是简单的“两个线程同时改一个 map”，而是“一个线程 miss 后去远端/数据库慢慢加载，另一个线程这时已经更新数据并失效缓存了”。加载过程会跨网络/回源 IO，不可能一直拿大锁把整个过程包住。要是这么做：

  * <font color=blue>锁粒度会非常粗</font>
    * 吞吐会明显下降
    * 还不适合跨节点场景

  举个你这个项目里的典型例子：

  1. 线程 A 读 key，缓存 miss，记下当前版本 v=5，开始去 peer 或数据源加载。
  2. 线程 B 更新了这个 key，对它执行失效，把版本推进到 6，见 src/geecache.cpp:101。
  3. 线程 A 这时才拿着旧值回来。
  4. 如果没有版本校验，A 可能把旧值重新写回缓存。
  5. 现在有了版本校验，A 会发现当前版本已经不是 5，于是丢弃这个旧值，见 src/geecache.cpp:198 和 src/geecache.cpp:246。

  所以结论是：

  - <font color=blue>**锁** 保护“数据结构安全”</font>
  - <font color=blue>**版本号** 保护“业务语义正确”</font>
  - 两者不是替代关系，而是互补关系

## 为什么当前用静态 peer 列表，不用注册中心？

> 1. 没有那么高的需求
> 2. 静态列表更容易保证“所有节点看到同一份成员视图”
> 3. 成员变更不是完全不支持，知识没有自动发现

因为当前项目的目标和规模下，静态 peer 列表是更合适的工程折中，不是因为不知道注册中心，而是 <font color=blue>暂时没必要把控制面复杂度引进来。</font>

  先说当前实现是什么样。这个项目的节点成员关系是通过 HTTPPool::Set(peers) 注入的，它会据此重建一致性哈希环和 peer getter。读路径再根据这份本地 peer 视图做路由。也就是说，<font color=blue>现在是“静态配置 + 本地一致性哈希”，不是接一个外部注册中心做自动发现。</font>

为什么这里先不用注册中心，核心有 4 点。

  - <font color=blue>节点数量小、部署关系稳定，peer 集合通常是已知的。</font>这个项目更像一个分布式缓存组件原型，重点在一致性哈希、singleflight、
    失效广播和一致性控制，不在服务发现本身。
  - <font color=blue>静态列表更容易保证“所有节点看到同一份成员视图”。</font>一致性哈希最怕各节点拿到不同的成员列表，否则 owner 计算会漂移。用静态配置时，这件事更可控。
  - <font color=blue>当前已经有一层“可用性兜底”，不是完全僵死的写死配置。</font>peer 不健康时不会继续被选中，健康探测逻辑在 src/http.cpp:281，路由
    时也会跳过不健康节点，见 src/http.cpp:116。
  - <font color=blue>成员变更不是完全不支持，只是没有自动发现。</font>Set(peers) 本身就支持更新 peer 列表，测试里也验证了从两节点扩到三节点的更新
    流程，见 tests/http_test.cpp:376。

>   “虽然当前成员列表是静态配置的，但 peer 的可用性不是写死的。系统会用后台 /healthz 探测和真实请求结果维护每个 peer 的健康状态；路由时如果 owner peer 不健康，就不会继续把请求打过去，而是退回本地回源加载。所以它不是注册中心式的动态发现，但已经具备一层运行时可用性兜底。”

为什么不上注册中心，是因为它会 <font color=blue>额外带来一整套控制面问题</font>：

  - 要维护注册、摘除、续约、watch、失败恢复
  - 要处理注册中心本身的高可用
  - 要处理成员变更传播延迟，避免不同节点短时间看到不同 ring
  - 要重新考虑节点上下线时的一致性和流量切换策略

这些问题在生产系统里值得做，但对当前这个项目来说，复杂度收益比不高。现在用静态 peer 列表，已经能把核心缓存语义跑通，而且更容易讲清楚。

当然，它的限制也很明确：

  - 扩缩容不够自动化
  - 节点增删通常需要同步更新各节点配置
  - <font color=blue>更适合小规模、成员相对稳定的集群，不适合高频弹性伸缩场景</font>

## 如果节点动态扩缩容，怎么做？

**<font color=red>当前项目</font>**

现在这套代码其实已经支持“更新 peer 列表”，只是还不是自动发现：

1. 外部控制面生成新的 peer 列表。
2. 把同一份列表下发到所有节点。
3. 所有节点调用 `Set(peers)` 原子替换本地 ring。
4. 一致性哈希加虚拟节点会让只有一部分 key 发生重映射，不是全量抖动。

**<font color=red>工程化做法</font>**

如果要做成真正的动态扩缩容，一般会上 **注册中心**，比如 etcd / Consul / Nacos，流程通常是：

1. 节点启动后注册自己，并持续心跳。
2. 所有节点 watch 成员列表变化。
3. 收到变更后，按同一个 membership version 重建本地一致性哈希环。
4. **扩容时，新节点先注册但不立刻接流量，等健康检查通过后再加入 ring。**
5. **缩容时，先把节点标成 draining，停止给它分配新 key，等在途请求和短期热数据访问过去后再摘除。**
6. 由于这是内存态 cache-aside 缓存，严格来说不一定要做数据迁移；新 owner 可以直接冷启动 miss 后回源加载。只是这样会有冷启动抖动，所以热点场景下可以加预热。

这里有两个关键 tradeoff：

- 如果你追求实现简单：只做“成员变更 + ring 重建”，不做缓存迁移。优点是逻辑简单，缺点是扩缩容时会有一段 cold miss。
- 如果你追求平滑：要加 warmup/drain，甚至做热点 key 预热或旧 owner 短期兜底，复杂度会明显上升。

## 如果广播失效很多，会不会影响性能？

当前实现的失效广播是同步 best effort fan-out。每次写成功后都会串行通知所有 peer，所以节点数多或更新频繁时，写延迟和网络开销都会上升。尤其当前广播不会过滤不健康节点，坏节点还可能把写路径拖慢到秒级。

进一步优化的话，我会优先考虑异步化、批量合并、跳过不健康 peer，以及 <font color=blue>用 MQ/Outbox 把广播从主写路径里拆出去</font>。”

## 项目中的难点

### 1. 分布式缓存一致性与防范“旧值回填”

- **场景痛点**：最大的难点是一致性，这不在于“删缓存”本身，而是“旧值已经在路上”。典型场景是：节点 A 正在从 owner B 读数据，此时节点 C 同时更新了同一个 key 并广播失效。如果只做简单的删除缓存，A 晚一点收到旧值后，还是可能把旧值重新写回本地缓存。
- **解决方式**：采用 Cache Aside 写路径，更新后广播失效本地和远端缓存。同时，我把版本推进逻辑放进 `src/geecache.cpp:143`，然后在远端读取回填和本地回源回填前，都做了 `src/geecache.cpp:291` 的版本校验，真正拦住“旧值回填”。
- **核心思考与创新**：我的思考是，问题不在于“缓存里有没有旧值”，而在于“网络和回源路径上还有没有旧值”。这里的创新点在于，版本校验同时覆盖了 main_cache 和 hot_cache，而不是仅仅保护 owner 节点上的主缓存。

### 2. 分布式路由与节点协作（防多跳循环）

- **场景痛点**：因为这是应用内缓存，不是一个中心化的缓存服务，所以必须解决“同一个 key 到底哪个节点负责”的问题。同时，要绝对避免节点之间出现 A 找 B、B 再继续转发这种多跳和循环问题。
- **解决方式**：用一致性哈希和虚拟节点机制把 key 映射到 owner 节点（在 `src/http.cpp:61` 构建哈希环，在 `src/http.cpp:108` 选 owner），然后通过 HTTP 去访问远端节点。为了防多跳，我在 peer 请求里加了 `X-Geecache-Peer` 头，服务端看到后直接走 `src/http.cpp:207`，到达 owner 后只做本地处理。
- **核心思考与创新**：这里的思考是把“路由决策”和“通信协议”分开：一致性哈希负责决定找谁，HTTP 负责怎么找它。创新点其实不在协议本身，而在于 owner-only 处理这层约束，这让整个节点协作链路变得更简单、更可控。

### 3. 热点 key 的高并发访问与流量放大

- **场景痛点**：多个线程、甚至多个节点同时访问同一个热点 key 时，如果本地缓存都 miss，就可能同时打到远端 owner 或底层数据源，造成缓存击穿和跨节点流量的放大。
- **解决方式**：我用 `src/singleflight/singleflight.cpp:6` 合并同一 key 的并发加载，只让一个请求真正执行回源；同时本地缓存层用 `std::shared_mutex` 做并发安全封装（见 `src/cache.h:15`）。在此基础上，我又加了 `src/geecache.cpp:258`，设计了 hotCache，对远端读回的数据做概率性本地副本回填。
- **核心思考与创新**：这里的思考是，仅仅让 owner 节点有缓存是不够的，因为非 owner 节点上的热点访问仍然会不断穿透网络。创新点就在于用这个很轻量的概率性副本机制，减少了后续重复的跨节点访问，改善非 owner 节点的本地命中率。

## 为什么一致性哈希这里用的是 crc32？如果 hash 分布不好会怎样？

这里选 crc32，本质上是一个很务实的工程取舍。

在这个项目里，默认哈希函数定义在 consistenthash.cpp。之所以用 crc32，主要有几个原因：

- 它足够快。缓存路由是高频路径，哈希不能太重。
- 它是确定性的。相同 key 在所有节点上必须算出同一个结果。
- 实现简单，直接依赖 zlib 就能用，项目本身也已经链接了 ZLIB。

> zlib 是一个很常用的 C 语言压缩库，主要提供两类能力：
>
> - **压缩/解压**：比如 deflate、gzip 相关能力
> - **校验算法**：比如 crc32
>
> 在你这个项目里，zlib 不是拿来压缩缓存数据的，而是拿来用它提供的 crc32 函数做一致性哈希，位置在 consistenthash.cpp。
>
> 对应这段逻辑本质上是：
>
> - 把字符串 key 转成字节序列
> - 调用 crc32(...)
> - 得到一个 uint32_t 哈希值
> - 再把这个值放到一致性哈希环里
>
> 项目的 CMake 里也显式依赖了 ZLIB，见 CMakeLists.txt。
>
> 如果面试里你想答得简洁一点，可以这么说：
>
> “zlib 是一个基础库，最常见用途是压缩和校验。我这个项目里主要用的是它的 crc32 实现，作为一致性哈希的默认哈希函数。”

也就是说，<font color=blue>这里并不是因为 crc32 最强，而是因为它满足“一致性哈希最基本的需求”：快、稳定、实现成本低。</font>

不过你要注意，一致性哈希对 hash 分布其实挺敏感的。如果 hash 分布不好，会直接带来这些问题：

- 节点负载不均。环上某些区间特别大，更多 key 会落到少数节点上。
- 热点更集中。本来热点 key 就可能压某个 owner，分布不好会雪上加霜。
- 虚拟节点效果变差。如果 hash 不均匀，虚拟节点也可能扎堆，没法真正把数据打散。
- 哈希冲突会让有效虚拟节点变少。这里底层是 std::map<uint32_t, std::string> ring_，如果多个虚拟节点算出同一个哈希值，后插入的会覆盖前面的，等于丢了副本位置，见 consistenthash.h。
- 扩缩容后的数据分布也会更难看。理论上“一致性哈希只迁移局部 key”这个性质还在，但迁移后的新分布可能仍然很偏。

所以你可以把它理解成：

<font color=blue>“一致性哈希保证的是节点变化时 key 不会大规模重映射，但它不保证分布一定均匀。分布均不均匀，很大程度取决于 hash 函数和虚拟节点设计。”</font>

面试里如果被追问“那生产环境还会不会用 crc32”，比较成熟的回答是：

- 小项目、教学项目、节点数不大时可以用，简单够用。
- 更看重分布质量时，通常会考虑 MurmurHash、xxHash，或者像 Ketama 那样用更稳定的散列方案。
- 也可以把 uint32_t 提升到 uint64_t，进一步降低冲突概率。

一句话总结就是：

crc32 在这里是“够用且便宜”的选择；如果 hash 分布不好，最终会表现为环不均匀、节点负载倾斜、虚拟节点失效、热点放大。

## 这套方案为什么只能做到最终一致，而不是强一致。

<font color=blue>因为这套设计的本质是“**DB/后端是真值源，缓存是异步失效的副本**”，而不是把所有读写都串行化到一个全局一致协议里。</font>

在这个项目里，写路径是 Set -> 先写后端 -> 再 Invalidate 本地和各个 peer，见 geecache.cpp。这意味着只要“写后端成功”和“所有节点都删掉旧缓存”之间存在时间差，就一定存在脏读窗口：

- 某个节点可能在广播失效到达前，仍然命中旧的 main_cache 或 hot_cache
- 某个 peer 失效请求可能暂时失败，代码里也是 best-effort，见 geecache_test.cpp
- 节点本地还允许 hot_cache 副本存在，所以同一个 key 可能在多个节点上短暂共存旧值

它做了版本号防护，确实很重要：并发加载旧值时，如果期间发生了失效，就不允许旧值重新回填缓存，见 geecache.cpp。但这个机制只能防“**旧值回填**”，不能防“**失效前那一瞬间已经读到旧缓存**”。所以它缩短了不一致窗口，却没有消灭窗口。

要做到强一致，通常至少得有这些更重的机制之一：

- 所有读写都经过同一个线性化 owner，并且 owner 切换也要一致
- 写入时阻塞读，直到所有副本确认失效
- 用分布式事务、共识协议、quorum 或全局版本序列化

而这个项目都没有做。它优先的是低延迟和高可用：本地先读、peer 不通可回源、失效是广播式传播、缓存是惰性回填。这些选择天然更接近“最终一致”。

面试里可以直接这样答：

“因为它是 cache-aside + 分布式本地副本缓存模型，写后只能异步广播失效，无法保证所有节点在同一时刻同时看到新值。版本号只能防止旧值回填，不能消除失效传播前的脏读窗口，所以它只能做到最终一致，而不是强一致。”

## 你是怎么验证“版本校验真的防住了旧值回填”的？测试设计思路是什么？

我是通过一个 **<font color=blue>刻意构造的并发时序测试</font>** 来验证的，核心测试就在 geecache_test.cpp 里的 VersionGuardPreventsStaleBackfill。

这个测试的设计目标不是“看起来像会出问题”，而是**强行把最危险的竞态窗口做出来**：

1. 先让一个读请求进入 getter，读到旧值 "old"，但先不要返回。
2. 在它卡住的这段时间里，主线程把底层数据改成 "new"，然后调用 Invalidate("Tom")。
3. Invalidate 会推进这个 key 的 version，并清掉本地缓存，见 geecache.cpp。
4. 然后再放开之前那个被卡住的旧读，让它带着旧值返回。
5. 如果没有版本校验，这个旧值就会被重新写回缓存，形成“旧值回填”。
6. 但现在实现里会在回填前比较版本；发现版本已经变了，就丢弃这次旧结果并重试加载，最终返回 "new"。

所以这个测试验证的不是“缓存最后是不是新值”这么简单，而是验证：

- 旧读确实先拿到了旧快照
- 失效确实发生在旧读返回之前
- 旧读返回后没有成功污染缓存
- 系统最终重新加载了新值，而不是把旧值塞回去

测试里用了 mutex + condition_variable 人工编排顺序，这一点很关键。因为并发 bug 如果只靠自然调度，测试会非常不稳定；这里是 **<font color=blue>手动把时序钉死</font>**，保证每次都稳定命中那个竞态窗口。

## TTL 过期删除是后台定时扫描，还是惰性删除？为什么？

这里是 **<font color=blue>惰性删除</font>**，不是后台定时扫描。

实现就在 cache.h。<font color=24b3aa>ConcurrentCache::Get 在读取时会先 Peek 一下当前条目，看是否过期：</font>

- 没过期，就正常返回并更新 LRU
- 过期了，就升级到写锁，把这个 key 从 LRU 里删掉，再返回 miss

也就是说，**只有当这个 key 再次被访问时，系统才会发现它过期并删除它**。当前项目里没有单独的后台线程去周期性遍历缓存做清理。

为什么这么做，主要是这几个考虑：

- 实现简单。这个项目重点是缓存主链路、分布式协作和并发控制，不想把复杂度花在后台清理器上。
- 开销更低。后台扫描需要周期性遍历缓存，会带来额外 CPU 和锁竞争。
- 对很多缓存场景够用。过期 key 只要在下一次访问时不再返回旧值，语义上通常已经满足 TTL 的核心目标。

但它也有明显代价：

- 冷 key 过期后如果再也不访问，会一直占内存，直到被 LRU 淘汰或整个缓存被清掉。
- 所以 TTL 在这里更准确说是“读取一致性控制 + 辅助回收”，不是严格的内存回收机制。

## 远端 peer 请求到了 owner 节点后为什么不能继续转发？怎么避免多跳和循环？

这里不能继续转发，核心原因是：**一旦一个请求已经被判定为“跨 peer 转发请求”，到达目标节点后就必须在本地终结**，否则会出现多跳、重复路由，甚至循环。

这套项目里的做法很明确，在 http.h 里定义了一个请求头 X-Geecache-Peer，在 http.cpp 里远端调用时会带上这个 header。owner 节点收到后，会判断这是 peer 请求，于是调用 group->GetLocallyOwned(key)，而不是普通的 group->Get(key)。GetLocallyOwned 会走 GetInternal(..., kLocalOnly)，明确禁止再次挑 peer，见 geecache.h 和 geecache.cpp。

为什么要这样设计，可以从三点讲：

- **避免多跳**：A 把请求转给 B 后，如果 B 还能再转给 C，请求链路就会变成 A -> B -> C，网络开销和延迟都会放大，缓存组件会越来越像一个不受控的转发网络。
- **避免循环**：如果节点之间的 peer 列表、健康状态、拓扑更新时机不完全一致，A 认为 owner 是 B，B 又可能因为本地视图不同再转回 A，或者转给别的节点，最终形成环路。
- **保证 owner 语义清晰**：一致性哈希的目的就是让某个 key 有一个明确 owner。既然请求已经到 owner 了，owner 就应该只做两件事：查本地缓存，或者自己回源；而不是继续把责任甩出去。

所以这个项目实际上的防护机制就是：

- client 节点第一次选 peer 时才允许转发
- peer 间转发请求打上 X-Geecache-Peer
- 接收方一旦发现这是 peer 请求，就只允许本地处理，不再走 PickPeer

## 为什么选择“更新后失效”而不是“更新时直接改缓存”？为什么 是更新后失效，而不是失效后更新？

这是缓存设计里很经典、也很容易被追问的一题。我们一起把它拆开。

**为什么选择“更新后失效”，而不是“更新时直接改缓存”**

这个项目本质上走的是 cache-aside 思路：**后端数据源是真值，缓存只是副本**。所以写路径更稳的做法是：

1. 先写后端
2. 成功后删缓存
3. 下次读再按新值回填

而不是“写后端的同时把所有缓存都改成新值”。

原因主要有这几个：

- **分布式下改缓存太难做全**。这里不是一个中心缓存，而是每个节点都有 main_cache，还有可能有 hot_cache 副本。你要“直接改缓存”，就得把所有节点上这个 key 的副本都找出来并改掉，工程复杂度很高。
- **双写很容易出顺序问题**。比如 DB 已经更新成功，但某个节点缓存更新失败；或者有些节点更新到了，有些没更新到，状态比“删掉再懒加载”更难判断。
- **删除比更新更幂等**。缓存更新失败时，你很难知道缓存里到底是什么值；但删除失败至少语义明确，可以重试。
- **适合热点读多写少场景**。缓存系统通常读多写少，写后删缓存，让下一次读自然回填，整体更符合这种业务模型。

所以一句话讲就是：

**<font color=blue>直接更新缓存属于分布式双写，复杂且脆弱；更新后失效把写路径收敛成“只保证真值源正确”，缓存交给读路径重建，更稳。</font>**

**为什么是“更新后失效”，而不是“失效后更新”**

顺序更关键。正确顺序是：

1. 先更新 DB / 后端
2. 再失效缓存

而不是先删缓存再写 DB。

因为如果你先失效，再更新后端，中间会出现一个很危险的窗口：

- 线程 A 先把缓存删了
- 线程 B 这时来读，发现缓存 miss
- B 去后端读，但此时后端还没更新完成，读到的还是旧值
- B 又把旧值重新写回缓存

这样你刚删掉的旧缓存，就又被“复活”了。

这正是这个项目为什么要加 **版本校验防旧值回填**，见 geecache.cpp 和对应测试 geecache_test.cpp。但注意，版本校验是在减小这种并发窗口带来的伤害，不代表“先失效后更新”本身是个好顺序。正确的基本策略仍然应该是：

- **先把真值源更新成功**
- **再广播失效缓存**

这样至少在缓存被删掉之后，后续回源读到的就是新值。

## 健康检查是怎么实现的？为什么“业务返回错误”不能直接判定 peer 不健康？

我们一起看这块的设计，它其实分成两层：**连通性健康** 和 **业务结果正确性**。这两个不能混为一谈。

**健康检查是怎么实现的**
实现集中在 http.h 和 http.cpp。

<font color=blue>当前 HTTPPool 会起一个后台线程 HealthCheckLoop，周期性做这几件事：</font>

1. <font color=blue>遍历所有远端 peer</font>
2. <font color=blue>对每个 peer 发一个 HTTP GET 到 /_geecache/healthz</font>
3. <font color=blue>如果返回 200，就把这个 peer 标记为健康</font>
4. <font color=blue>如果请求失败或超时，就按规则更新健康状态</font>

具体代码路径是：

- 后台线程入口：HTTPPool::HealthCheckLoop
- 单个探测：HTTPPool::CheckPeerHealth
- 状态更新：UpdatePeerHealth / UpdatePeerHealthFromProbe

这里探测的不是业务 key，而是一个专门的健康检查接口：

- 路径：/_geecache/healthz
- 成功条件：HTTP 200
- 超时设置：连接和读超时都比较短，当前是 200ms，见 http.cpp

另外，正常 peer 请求过程中也会顺手更新健康状态：

- 如果 HTTP 请求根本发不出去，比如连接失败、超时，HttpGetter::Get / Invalidate 会把 peer 标成不健康
- 如果请求能到达并拿到 HTTP 响应，就会标成健康

也就是说，当前项目判断的“健康”，本质上是：

- 这个 peer 的 HTTP 服务是否可达
- 它是否还能正常响应缓存节点间协议

**为什么“业务返回错误”不能直接判定 peer 不健康**
因为“业务错误”通常表示的是**请求内容或数据状态有问题**，不是**节点本身挂了**。

比如这些都属于业务错误，但不说明 peer 不健康：

- key 不存在
- group 不存在
- 本地回源失败
- owner 节点应用逻辑报错
- protobuf 解析前的业务层返回 500

这些情况说明的是“这次请求没拿到你想要的数据”，但节点本身依然是活着的、网络是通的、HTTP 服务也是正常的。
如果把这种错误直接判成“不健康”，会有几个问题：

- **误摘流量**：一个其实活着的节点会被错误踢出路由
- **放大故障**：本来只是某个 key 查不到，结果变成整台节点都不再被访问
- **掩盖真实问题**：数据缺失和网络故障是两类问题，混在一起会让排障很乱
- **造成路由震荡**：某些瞬时业务错误可能让节点一会健康一会不健康

这个项目的测试专门验证了这一点，见 http_test.cpp 里的 PeerApplicationErrorsDoNotMarkPeerUnhealthy。测试思路很直接：

- peer 节点服务是正常启动的
- 但它对某个 key 返回业务错误
- 调用方这次请求虽然失败了，但 IsPeerHealthy(...) 仍然应该保持 true

这说明项目明确区分了：

- **传输层/服务可用性错误**：可以判 unhealthy
- **应用层业务错误**：不能直接判 unhealthy

## 缓存容量怎么控制的？为什么按字节限流？

这个项目的容量控制是按字节数做的，不是按 key 数量。Group 里主缓存和 hot cache 分别有自己的容量上限，底层用 LRU 维护当前占用；每次写入后如果超出 max_bytes，就持续淘汰最久未使用的数据，直到回到容量范围内。TTL 只是惰性过期，不是核心容量控制手段。

按字节数比按 key 数量更合理，核心原因是：**缓存真正稀缺的资源是内存，不是 key 的个数。**

按 key 数量限制不符合真实资源模型，因为不同 key/value 的大小差异很大。缓存真正要控制的是内存占用，所以按字节数更合理，至少能让大对象更早触发淘汰，避免少量大 value 把缓存撑爆。

## 节点挂了访问会发生什么？

如果节点 A 挂了，我们可以分两类流量来看，这样最清楚。

**1. 请求本来就是直接打到 A 的**
那最直接的结果就是：**这次请求在入口层就失败**。

<font color=blue>因为 A 的进程已经不在了，或者 HTTP 服务不可达了，所以：</font>

- 如果前面有负载均衡，LB 通常会把 A 摘掉，后续新请求不再转发给它
- 如果客户端还是直连 A，那就是连接失败、超时或 5xx

这部分不是 geecache 内部在“接管”，而是网络入口层面的失败。

**2. 请求打到别的节点，但某些 key 的 owner 恰好是 A**
这才是 geecache 内部真正处理的场景。

比如请求先到 B，B 发现这个 key 的 owner 是 A。正常情况下，B 会去找 A；但如果 A 已经被健康检查标记为不健康，或者远端请求直接失败，那么：

- B 不会继续把请求发给 A
- 也不会路由到“下一个健康节点”
- 而是退化成：
  - 先看自己本地有没有 main_cache / hot_cache
  - 没有的话就调用本地 getter 回源（<font color=24b3aa>原本这个数据应该经过一致性哈希路由到 A 来拿，但现在 A 挂了，只能 B 自己去拿了，也因此，某个节点挂了，不应影响整体的可用性，只是访问可能变慢了。不过这个结论有一个重要前提：各节点必须还能访问同一个共享真值源。如果不是这样，比如：某些数据只有挂掉的那个节点能拿到或者当前节点自己的 getter 根本拿不到这个 key，那节点挂了之后，这部分请求就会直接失败，而不只是变慢。</font>）

对应逻辑在 http.cpp 的 PickPeer 和 geecache.cpp 的 Load。

**所以实际效果取决于部署前提**

有三种典型结果：

- B 本地有 hot_cache：还能直接返回，用户几乎无感
- B 没有缓存，但 B 能访问共享后端：还能返回，只是性能变差、后端压力上升
- B 既没有缓存，也不能自己回源：这次请求就失败

**写和失效也会受影响**

如果这时有某个 key 做 Invalidate，广播到 A 会失败。当前实现是 best-effort：

- 其他活着的节点照常失效
- A 因为挂了收不到这次失效
- 错误会被汇总返回，但不会阻塞整个流程

## 怎么检测挂了的节点

这个项目里，“节点挂了”的处理不是主从切换，而是**<font color=blue>健康探测 + 读路径降级</font>**。

我们一起顺着代码看一下。

HTTPPool 会启动一个后台健康检查线程，周期性访问每个 peer 的 /_geecache/healthz 接口，见 http.cpp。

如果某个 peer：

- 连接失败
- 超时
- 健康检查拿不到 200

它就会被标记为 unhealthy。

<font color=24b3aa>另外，正常的 peer 请求如果网络层失败，HttpGetter::Get / Invalidate 也会顺手把这个 peer 标成不健康。</font>

## 挂了的节点是怎么处理的？

当前实现里，**挂掉的节点会继续留在 peer 列表和一致性哈希环里**，不会自动移除。

**现在实际怎么处理**

- 健康检查会把这个节点标记成 unhealthy
- PickPeer(key) 如果发现 owner 是这个不健康节点，会直接放弃远端访问
- 然后读请求退化成本地缓存命中或本地 getter 回源

也就是说，它做了的是：

- **摘流量**
- **不摘成员**

这样做会有两个重要后果：

- 一致性哈希环里仍然包含这个节点，所以某些 key 还是会映射到它
- 但因为它不健康，PickPeer 不会真的把请求发过去，于是这些 key 会频繁走降级路径

如果这个节点恢复了：

- 健康检查会重新探测到它可用
- 它会重新变成 healthy
- 请求又会重新路由到它
- 不需要重建 ring，只要它还在 peer 列表里就行

## 如果数据量特别大，版本表 versions_ 会不会膨胀？如果网络分区了怎么处理？

versions_ 是这个项目里为了**防旧值回填**专门加的一层本地元数据，不是业务数据本身。

**versions_ 是什么**
它定义在 geecache.h 里，是：

- std::unordered_map<std::string, uint64_t> versions_

含义可以理解成：

- key -> 当前这个 key 已知的最新失效版本号

配套还有一个全局递增时钟 g_version_clock，定义在 geecache.cpp。

整个机制是这样工作的：

1. 某个 key 被 Invalidate 时，会先 NextVersion() 生成一个更大的 version
2. 然后 ApplyInvalidation(key, version) 把这个 version 写进 versions_[key]
3. 同时清掉 main_cache 和 hot_cache
4. 之后如果有一个旧的加载结果晚到了，在准备回填缓存前，会比较它加载开始时看到的 version 和当前 version 是否一致
5. 如果不一致，说明期间发生过更新/失效，这个旧值就不能写回缓存

所以它的职责不是存数据，而是存**“这个 key 有没有在我加载期间被更新过”**的判定依据。

**versions_ 会不会膨胀**
会，而且这是当前实现里一个很真实的风险点。

原因很直接：

- versions_ 是按 key 存的
- 只有在 Invalidate / ApplyInvalidation 时才写入
- 当前代码里**没有清理机制**
- key 即使已经不在缓存里了，对应的版本记录也不会自动删除

所以如果：

- key 空间很大
- 更新/失效很多
- 长期运行

那 versions_ 可能会持续增长，最终变成一张只增不减的 map。

更准确地说，它的增长速度主要取决于：

- 被失效过的不同 key 数量
- 不是被读取过的 key 数量

因为普通读不会往 versions_ 里写。

**怎么改进**
如果面试官追问优化方向，你可以给几个成熟一点的方案：

- **和缓存项生命周期绑定**
  把 version 放进 cache entry 或单独的 key metadata，随着缓存项淘汰一起清理。
- **加 TTL / 老化回收**
  对长时间没再失效、也没访问的 version 记录做过期删除。
- **分代 / epoch 机制**
  对 peer 拓扑变更或批量失效用 epoch，而不是每个 key 永久记一条。
- **后台清扫**
  定期扫描 versions_，删除既不在 cache、又长期没访问的 key 版本。

**如果网络分区了怎么处理**
当前实现里，**没有专门的网络分区处理协议**。它做的只是：

- 健康检查失败后把对端标成 unhealthy
- 读请求不再发给这个 peer
- 退化成本地缓存命中或本地 getter 回源
- 失效广播对打不通的 peer 会失败，并把错误返回出来

也就是说，它没有：

- quorum
- 共识协议
- 重试队列
- 反熵同步
- 恢复后的版本对账

所以网络分区下会发生什么？

**分区期间**

- 连不通的节点收不到 Invalidate
- 它本地的 versions_ 不会推进
- 它的旧 main_cache / hot_cache 可能继续存在
- 如果客户端还能访问这个隔离侧节点，它可能继续返回旧值

**恢复之后**

- 健康检查会把节点重新标成 healthy
- 但当前实现**不会自动补发分区期间丢失的失效消息**
- 也不会做版本同步

所以恢复后可能出现：

- 某些节点已经是新值
- 某些节点还留着旧缓存
- 只能依赖后续再次失效、TTL 过期、LRU 淘汰，或者重新加载慢慢收敛

这也是为什么这套方案只能讲“最终一致”，而且在网络分区场景下，这个“最终”其实不够强，因为它缺少恢复后的补偿机制。

## 一次 Get(key) 请求从进来到返回，完整链路是什么？三种路径（本地、远端、回源）怎么走？

一次 Get(key) 先统一走 GetInternal，优先查本地 main_cache 和 hot_cache；本地 miss 后进入 Load，由 singleflight 防止并发击穿。之后有三条路：本地命中就直接返回；如果 key 归远端 owner 且对端健康，就走 peer 请求，从 owner 拉取并可能写入本地 hot cache；如果不能走远端，就调用本地 getter 回源，并把结果写入 main cache。整个加载过程都受 version 校验保护，避免失效期间的旧值重新回填。
